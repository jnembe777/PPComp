

# Projet : Codage LMD Versatile pour Classes de Processus Ponctuels

## 1. Objectif du Projet
Développer un framework de compression de données basé sur le principe de la **Longueur Minimale de Description (MDL/LMD)** appliqué à diverses classes de processus ponctuels. Le système doit être capable de :
1.  Choisir automatiquement la **représentation** la plus efficace des données (versatilité).
2.  Estimer la **fonction d'intensité** optimale parmi plusieurs familles de fonctions.
3.  Comparer la performance avec un **codage uniforme** (cas de référence sans estimation).

## 2. Cadre Mathématique Général

Soit $(\Omega, \mathcal{F}, P)$ un espace de probabilité. On observe un processus ponctuel marqué $N$ sur un intervalle $[0, T]$.
Un réalisé du processus est noté $\omega = \{(\tau_k, m_k)\}_{k=1}^N$, où $\tau_k$ sont les temps de saut et $m_k \in \mathcal{E}$ les marques (états, couleurs, etc.).

### 2.1. Modèle d'Intensité Multiplicative (Aalen)
Comme décrit dans `Uniform_temporal.tex` et `versatile_main.tex`, l'intensité stochastique $\lambda(t)$ est modélisée par :
$$ \lambda(t) = \alpha(t)Y(t) $$
où :
*   $\alpha(t)$ est la fonction d'intensité de base (inconnue, à estimer).
*   $Y(t)$ est un processus prévisible observable (ex: individu à risque).

La vraisemblance pour un paramètre $\theta$ (définissant $\alpha$) est donnée par (cf. `versatile_main.tex`) :
$$ L_{\theta}(N) = \exp \left( \int_0^T \log(\alpha_{\theta}(s)) dN(s) - \int_0^T \alpha_{\theta}(s)Y(s) ds \right) $$

## 3. Versatilité des Représentations
Avant tout codage, le système doit choisir la représentation des données la plus compacte. Selon `versatile_pp.tex`, quatre représentations sont disponibles :

1.  **Séries Temporelles (Time Stamps) :** Liste exacte des temps $\{\tau_1, \dots, \tau_N\}$.
    *   *Usage :* Précision temporelle fine.
2.  **Comptage (Histogrammes) :** Vecteur de comptes dans des bins de temps fixes $\{n_1, \dots, n_r\}$.
    *   *Usage :* Analyse de taux, données discrétisées.
3.  **Intervalles (Inter-event times) :** Liste des durées entre sauts $\{\Delta_1, \dots, \Delta_N\}$.
    *   *Usage :* Régularité des événements, processus de renouvellement.
4.  **Binaire (Boolean Vector) :** Vecteur de bits indiquant la présence d'un saut dans chaque bin $\{b_1, \dots, b_r\}$.
    *   *Usage :* Détection d'événements, perte de précision temporelle acceptée.

**Critère de choix :** La représentation $R \in \{TimeStamp, Count, Interval, Binary\}$ est choisie pour minimiser la longueur de code totale estimée.

## 4. Estimation de l'Intensité (Familles de Fonctions)
Pour le codage LMD non-uniforme, on estime $\alpha(t)$ dans une famille de fonctions $\Gamma_n$. Selon votre demande et le contexte d'estimation non-paramétrique de `versatile_main.tex`, les familles candidates sont :

1.  **Histogrammes (Step functions) :** $\alpha(t) = \sum c_i \mathbb{I}_{[t_i, t_{i+1}]}(t)$.
2.  **Splines :** Fonctions polynomiales par morceaux lisses.
3.  **Polynômes :** $\alpha(t) = \sum_{k=0}^d a_k t^k$.
4.  **Ondelettes (Wavelets) :** Décomposition multi-résolution pour capturer les singularités locales.
5.  **Polynômes Trigonométriques :** $\alpha(t) = \sum (a_k \cos(kt) + b_k \sin(kt))$ (pour données périodiques).

## 5. Critère LMD (Minimum Description Length)
Le cœur du projet repose sur la minimisation de la complexité stochastique. Selon `versatile_main.tex` et `uniform_temp_coding.tex`, l'estimateur LMD $\hat{P}_n$ est défini par :

$$ \hat{P}_n = \arg \min_{Q \in \mathcal{P}} \left( -\log Q(X^n) + C_n(Q) \right) $$

Où :
*   $-\log Q(X^n)$ est la longueur de code des données给定 le modèle (négatif de la log-vraisemblance).
*   $C_n(Q)$ est la pénalité de complexité du modèle (nombre de paramètres, dimension de la famille, etc.).

**Bornes de Convergence :**
La performance est mesurée via la distance de Hellinger $H^2$. D'après `versatile_main.tex`, l'estimateur satisfait presque sûrement :
$$ H^2(P, \hat{P}_n) \leq \min_{Q \in \Gamma_n} \left[ H^2(P, Q) + \frac{1}{n} C_n(Q) \right] $$

## 6. Classes de Processus Ponctuels Spécifiques
Le système doit implémenter les vraisemblances spécifiques pour chaque classe (cf. `versatile_main.tex`) :

1.  **Processus Ponctuel Réel (Standard) :**
    *   Intensité scalaire $\lambda(t)$.
    *   Vraisemblance de Poisson non-homogène.
2.  **Processus Ponctuel Vectoriel (Multivarié) :**
    *   Vecteur d'intensités $\boldsymbol{\lambda}(t) = (\lambda_1(t), \dots, \lambda_d(t))$.
    *   Prend en compte les dépendances entre composantes.
3.  **Processus Ponctuel Spatial :**
    *   Intensité $\lambda(s)$ où $s \in \mathbb{R}^d$.
    *   Utilisation de noyaux ou de discrétisation spatiale.
4.  **Processus Markovien des Transitions :**
    *   Modèle de type Aalen-Johansen.
    *   Intensités de transition $\alpha^{hj}(t)$ entre états $h$ et $j$.
    *   Vraisemblance : $L = \prod_{h,j} \exp \left( \int (1-\gamma^{hj})Y^h ds + \int \log \gamma^{hj} dN^{hj} \right)$.

## 7. Codage Uniforme (Cas de Référence)
Comme décrit dans `uniform_temp_coding.tex` et `Uniform_temporal.tex`, le codage uniforme correspond à l'absence d'estimation d'intensité (hypothèse d'uniformité sur $[0, T]$ et sur les marques).

Les longueurs de code de référence sont :
1.  **Encodage État Complet :** $L_1 = r \log m$ (où $r$ bins de temps, $m$ états).
2.  **Liste des Sauts :** $L_2 = \log N + N \log(rm)$.
3.  **Vecteur Booléen :** $L_3 = r + N \log m$.
4.  **Adresse Combinatoire (Optimal Uniforme) :**
    $$ L_4 = \log N + \log \binom{r}{N} + N \log m $$
    *C'est la borne inférieure pour le codage uniforme.*

**Règle de décision :** Le codage LMD (avec estimation) ne sera retenu que si sa longueur totale (Données + Modèle) est inférieure à $L_4$.

## 8. Algorithme Global du Projet

Voici la procédure étape par étape à implémenter :

1.  **Prétraitement :**
    *   Input : Données brutes (timestamps, marques).
    *   Discrétisation optionnelle (paramètre $r$).

2.  **Boucle sur les Représentations ($R$) :**
    *   Pour chaque $R \in \{TimeStamp, Count, Interval, Binary\}$ :
        *   Transformer les données selon $R$.

3.  **Boucle sur les Classes de Processus ($\mathcal{C}$) :**
    *   Pour chaque classe (Réel, Vectoriel, Markovien...) :

4.  **Boucle sur les Familles d'Estimation ($\mathcal{F}$) :**
    *   Pour chaque famille (Histogramme, Splines, Ondelettes...) :
        *   **Estimation :** Calculer $\hat{\alpha}$ maximisant la vraisemblance (ou minimisant le risque).
        *   **Complexité :** Calculer $C_n(\hat{\alpha})$ (ex: dimension du modèle $\times \log n$).
        *   **Code Données :** Calculer $-\log L_{\hat{\alpha}}(Data)$.
        *   **Code Total :** $L_{total} = -\log L_{\hat{\alpha}} + C_n(\hat{\alpha})$.

5.  **Comparaison avec Uniforme :**
    *   Calculer $L_{uniforme} = \min(L_1, L_2, L_3, L_4)$ selon les formules de `uniform_temp_coding.tex`.

6.  **Sélection Finale :**
    *   Choisir le triplet $(R^*, \mathcal{C}^*, \mathcal{F}^*)$ qui minimise $\min(L_{total}, L_{uniforme})$.

7.  **Codage :**
    *   Encoder les paramètres du modèle choisi.
    *   Encoder les résidus (données) utilisant le modèle choisi (codage arithmétique ou entropique).

## 9. Métriques de Performance
Pour valider le projet, les métriques suivantes seront calculées (cf. `uniform_temp_coding.tex` section Numerical results) :
*   **Longueur de Code Moyenne :** $E[L]$.
*   **Redondance :** Différence entre la longueur obtenue et l'entropie théorique.
*   **Taux de Compression :** Ratio entre taille brute et taille codée.
*   **Temps de Calcul :** Pour évaluer la faisabilité temps-réel.

## 10. Références Clés du Projet
*   **Modèle d'intensité & Vraisemblance :** `Uniform_temporal.tex`, `versatile_main.tex`.
*   **Formules de Codage Uniforme :** `uniform_temp_coding.tex`.
*   **Choix de Représentation :** `versatile_pp.tex`.
*   **Bornes Théoriques (Hellinger/MDL) :** `versatile_main.tex`.



# Projet : Codage Vidéo par Processus Ponctuels Monochromatiques (CPPM)

## 1. Modèle Mathématique

Ce modèle formalise la transformation d'une vidéo en une somme de processus ponctuels non-marqués, permettant d'estimer les intensités par couleur et d'appliquer le principe de Longueur Minimale de Description (MDL).

### 1.1. Formalisation de la Donnée Vidéo
Soit une vidéo définie sur un domaine spatial $\mathcal{S}$ (ensemble de pixels ou de blocs) et un intervalle de temps $[0, T]$.
Pour chaque source spatiale $i \in \{1, \dots, n\}$ (ex: un pixel), la couleur est une fonction $C_i(t) \in \mathcal{E}$, où $\mathcal{E} = \{c_1, \dots, c_m\}$ est l'espace des marques (palette de couleurs discrétisée, $m < \infty$).

### 1.2. Transformation en Processus Ponctuel Marqué
Classiquement, l'historique d'un pixel est un processus ponctuel marqué $N_i = \{(\tau_k, m_k)\}_{k=1}^{K_i}$, où $\tau_k$ sont les instants de changement de couleur et $m_k \in \mathcal{E}$ la nouvelle couleur.
La vraisemblance standard inclut un terme de codage des marques ($N \log m$).

### 1.3. Décomposition Monochromatique (Innovation)
Au lieu de coder un processus marqué, nous définissons **$m$ processus ponctuels simples (non-marqués)** par source spatiale, notés $N_i^{(c)}$ pour chaque couleur $c \in \mathcal{E}$.
*   **Définition :** $N_i^{(c)}(t) = \sum_{k} \mathbb{I}_{\{\tau_k \leq t, m_k = c\}}$.
*   **Propriété :** Un saut dans $N_i^{(c)}$ à l'instant $t$ implique implicitement que la couleur du pixel $i$ devient $c$ à l'instant $t$.
*   **Avantage Codage :** Le terme $N \log m$ (codage des marques) est supprimé. L'information de couleur est portée par l'indice du processus.

### 1.4. Modèle d'Intensité Multiplicative (Aalen)
Pour chaque processus monochromatique $N_i^{(c)}$, on assume un modèle d'intensité stochastique (cf. `Uniform_temporal.tex`, `versatile_main.tex`) :
$$ \lambda_i^{(c)}(t) = \alpha_i^{(c)}(t) Y_i^{(c)}(t) $$
où :
*   $\alpha_i^{(c)}(t)$ est l'intensité de base déterministe (inconnue, à estimer).
*   $Y_i^{(c)}(t)$ est un processus prévisible (ex: indicateur de risque, souvent $Y=1$ si le pixel est actif).

La vraisemblance pour une couleur $c$ sur l'ensemble des $n$ sources est (cf. `versatile_main.tex`) :
$$ L_{\alpha^{(c)}} = \prod_{i=1}^n \exp \left( \int_0^T \log(\alpha_i^{(c)}(s)) dN_i^{(c)}(s) - \int_0^T \alpha_i^{(c)}(s)Y_i^{(c)}(s) ds \right) $$

### 1.5. Critère de Sélection de Modèle (MDL)
Conformément à `versatile_main.tex`, l'estimateur $\hat{\alpha}$ est choisi pour minimiser la complexité stochastique :
$$ \hat{\alpha} = \arg \min_{\alpha \in \Gamma} \left( -\log L_{\alpha} + C_n(\alpha) \right) $$
où $C_n(\alpha)$ est la pénalité de complexité (nombre de paramètres, dimension de la famille de fonctions).

### 1.6. Représentations Versatiles
Pour chaque processus $N_i^{(c)}$, on choisit la représentation $R \in \{\text{Timestamp, Count, Interval, Binary}\}$ (cf. `versatile_pp.tex`) qui minimise la longueur de code.
Les longueurs de code de référence (codage uniforme) sont adaptées de `uniform_temp_coding.tex` pour la décomposition monochromatique :
1.  **Vecteur d'état (Binaires) :** $L_{bin} = \sum_{c=1}^m r$ (où $r$ est le nombre de bins temporels).
2.  **Liste de sauts (Sans marques) :** $L_{list} = \sum_{c=1}^m (\log N_c + N_c \log r)$.
3.  **Adresse Combinatoire (Optimal Uniforme) :** $L_{comb} = \sum_{c=1}^m (\log N_c + \log \binom{r}{N_c})$.

Le codage MDL non-uniforme sera retenu si :
$$ -\log L_{\hat{\alpha}} + C_n(\hat{\alpha}) < \min(L_{bin}, L_{list}, L_{comb}) $$

## 2. Cahier des Charges Informatique

### 2.1. Objectifs du Système
*   **Entrée :** Flux vidéo brut (séquence d'images).
*   **Sortie :** Flux binaire compressé (bitstream) et vidéo reconstruite (décompression).
*   **Contrainte Principale :** Exploiter la parcimonie temporelle (peu de changements de couleur par pixel) et la régularité des intensités par couleur.
*   **Performance :** Taux de compression supérieur aux codecs standards (type H.264/HEVC) pour les vidéos à faible mouvement ou à couleurs discrètes, avec possibilité de décision en temps réel.

### 2.2. Architecture Logicielle

Le système se décompose en 4 modules principaux :

#### Module 1 : Extraction et Discrétisation (Preprocessing)
*   **Fonction :** Transformer le flux vidéo en données de processus ponctuels.
*   **Tâches :**
    1.  **Quantification Colorimétrique :** Réduire l'espace colorimétrique continu à $m$ couleurs dominantes (ex: $m=16, 64, 256$). *Référence : `uniform_temp_coding.tex` (paramètre $m$).*
    2.  **Détection de Sauts :** Pour chaque pixel $(x,y)$, identifier les instants $t$ où $C(x,y,t) \neq C(x,y,t-1)$.
    3.  **Séparation par Canal :** Aiguiller chaque saut vers le processus $N^{(c)}$ correspondant à la nouvelle couleur.
    4.  **Discrétisation Temporelle :** Mapper le temps continu sur $r$ bins (ex: $r=256, 512, \dots$ par GOP - Group of Pictures).
*   **Output :** Structure de données $\{ N_i^{(c)} \}_{i,c}$.

#### Module 2 : Estimation et Sélection de Modèle (MDL Core)
*   **Fonction :** Choisir la meilleure stratégie de codage par bloc de pixels.
*   **Tâches :**
    1.  **Estimation d'Intensité :** Pour chaque couleur $c$, estimer $\hat{\alpha}^{(c)}(t)$ dans plusieurs familles (Histogramme, Splines, Polynômes, Ondelettes). *Référence : `versatile_main.tex`.*
    2.  **Calcul de Complexité :** Calculer $C_n(\hat{\alpha})$ (coût de description du modèle).
    3.  **Choix de Représentation :** Comparer les représentations (Timestamp vs Binary vs Count) pour chaque canal. *Référence : `versatile_pp.tex`.*
    4.  **Décision Uniforme vs MDL :** Appliquer l'Algorithme 1 de `uniform_temp_coding.tex` adapté pour comparer le coût MDL au coût uniforme $L_{comb}$.
*   **Output :** Métadonnées du modèle (type de fonction, paramètres, représentation choisie).

#### Module 3 : Codage et Compression (Encoder)
*   **Fonction :** Générer le bitstream.
*   **Tâches :**
    1.  **Codage des Paramètres :** Encoder les coefficients du modèle choisi (ex: coefficients de spline) using un code universel d'entiers (ex: code $\delta$ de Rissanen). *Référence : `uniform_temp_coding.tex` section Universal encoding.*
    2.  **Codage des Résidus :** Encoder les temps de sauts réels par rapport au modèle d'intensité (codage arithmétique ou entropique basé sur $\hat{\alpha}$).
    3.  **Multiplexage :** Assembler les flux des $m$ canaux de couleur et des $n$ sources spatiales.
*   **Output :** Fichier binaire compressé.

#### Module 4 : Décompression et Reconstruction (Decoder)
*   **Fonction :** Reconstruire la vidéo.
*   **Tâches :**
    1.  **Décodage des Métadonnées :** Lire le type de modèle et les paramètres.
    2.  **Génération des Sauts :** Reconstruire les temps de sauts $\tau_k$ pour chaque processus $N^{(c)}$.
    3.  **Reconstruction Temporelle :** Pour chaque pixel, fusionner les sauts des $m$ canaux pour reconstruire la chronologie des couleurs $C_i(t)$.
    4.  **Rendu :** Générer les frames vidéo.

### 2.3. Spécifications des Algorithmes Clés

#### Algorithme de Choix de Codage (Adapté de `uniform_temp_coding.tex`)
Pour chaque bloc de pixels et chaque couleur $c$ :
1.  Compter $N_c$ (nombre de sauts) et fixer $r$ (résolution temps), $m$ (nombre de couleurs).
2.  Calculer les seuils théoriques $\delta_{r,m}$, $\gamma_{34}$, etc. (cf. Proposition 1 de `uniform_temp_coding.tex`).
3.  **Si** $N_c$ est très élevé (proche de $r$) : Choisir **Codage Vecteur Booléen** ($L_3$).
4.  **Si** $N_c$ est faible : Choisir **Codage Combinatoire** ($L_4$) ou **MDL** si l'intensité est structurée.
5.  **Si** MDL choisi : Estimer $\alpha$, calculer $-\log L + C_n$, comparer à $L_4$. Garder le minimum.

#### Algorithme d'Estimation (Adapté de `versatile_main.tex`)
1.  Définir une suite de modèles emboîtés $\Gamma_1 \subset \Gamma_2 \dots$ (ex: histogrammes de 2, 4, 8 bins...).
2.  Pour chaque modèle, calculer la vraisemblance logarithmique.
3.  Ajouter la pénalité $C_n$ (proportionnelle à la dimension du modèle $\times \log n$).
4.  Sélectionner le modèle minimisant la somme.

### 2.4. Exigences de Performance
*   **Taux de Compression :** Viser un gain de 20% par rapport au codage uniforme optimal ($L_4$) sur les vidéos à faible entropie temporelle.
*   **Complexité Temporelle :** L'estimation d'intensité doit être linéaire $O(N)$ ou $O(r)$ pour permettre le temps réel.
*   **Robustesse :** Le système doit retomber sur le codage uniforme si l'estimation MDL échoue (sur-apprentissage).
*   **Scalabilité :** Le paramètre $m$ (couleurs) doit être ajustable dynamiquement selon le contenu (ex: vidéo noir/blanc vs couleur).

### 2.5. Livrables Attendus
1.  **Bibliothèque C++/Python :** Implémentation des modules d'extraction et de codage.
2.  **Moteur MDL :** Implémentation des estimateurs d'intensité (Splines, Histogrammes).
3.  **Outil de Benchmarking :** Comparaison des longueurs de code ($L_1, L_2, L_3, L_4, L_{MDL}$) sur des datasets de référence.
4.  **Documentation :** Spécification du format de fichier binaire (en-tête, métadonnées, flux de données).

## 3. Justification Théorique (Synthèse des Documents)

1.  **Efficacité du Codage Monochromatique :** En supprimant le codage explicite des marques ($N \log m$), on se ramène à coder des processus de comptage simples. Cela réduit la borne inférieure de l'entropie, surtout si certaines couleurs sont rares (intensité $\alpha^{(c)}$ faible).
2.  **Optimalité MDL :** Le théorème principal de `versatile_main.tex` garantit que l'estimateur $\hat{P}_n$ converge vers la vraie distribution $P$ avec un taux lié à la distance de Hellinger. Cela assure que le codage n'est pas seulement compressant, mais statistiquement consistant.
3.  **Versatilité :** Grâce à `versatile_pp.tex`, le système n'est pas figé sur une représentation (ex: timestamps). Il peut basculer vers une représentation binaire si l'activité est dense, garantissant une robustesse face à différents types de vidéos (animation vs film réel).
4.  **Bornes Uniformes :** Les formules de `uniform_temp_coding.tex` fournissent un "plancher" de performance. Le système MDL ne doit jamais faire pire que ce plancher (grâce à la clause de comparaison dans l'algorithme).

Ce cahier des charges fournit une base solide pour développer un codeur vidéo novateur basé sur la théorie des processus ponctuels, alliant rigueur mathématique et implémentation pratique.


# Projet : Codage LMD Versatile Multi-Classes pour Données Vidéo

## 1. Modèle Mathématique Unifié

Le cadre général repose sur le principe MDL décrit dans `versatile_main.tex`. L'objectif est de sélectionner le modèle $\hat{P}_n$ qui minimise la longueur de description totale :

$$ \hat{P}_n = \arg \min_{P \in \mathcal{M}} \left( -\log L_P(Data) + C_n(P) \right) $$

où $L_P$ est la vraisemblance du processus et $C_n(P)$ la complexité du modèle (pénalité).

### 1.1. Classes de Processus et Vraisemblances Spécifiques

Pour chaque type de processus, la fonction d'intensité $\lambda$ est estimée dans une famille de fonctions $\Gamma_n$ (Histogrammes, Splines, etc.).

#### A. Processus Ponctuel Réel avec Marques (Marked Point Process)
*   **Contexte Vidéo :** Historique d'un pixel. Les sauts sont les changements de couleur, la marque $m_k$ est la nouvelle couleur.
*   **Modèle d'Intensité :** $\lambda(t, m) = \alpha(t, m)Y(t)$.
*   **Vraisemblance (cf. `Uniform_temporal.tex` Eq. 2) :**
    $$ L_{\alpha}(N) = \exp \left( \sum_{k=1}^N \log \alpha(\tau_k, m_k) - \int_0^T \sum_{m \in \mathcal{E}} \alpha(s, m)Y(s) ds \right) $$
*   **Estimation :** On estime $\alpha(t, m)$ pour chaque couleur $m$ séparément ou conjointement.

#### B. Processus Ponctuel Vectoriel (Multivarié)
*   **Contexte Vidéo :** Canaux de couleur couplés (R, G, B) ou plusieurs pixels corrélés.
*   **Modèle d'Intensité :** Vecteur $\boldsymbol{\lambda}(t) = (\lambda_1(t), \dots, \lambda_d(t))$.
*   **Vraisemblance :** Produit des vraisemblances marginales si indépendants, ou modèle de copule si dépendants.
    $$ L_{\boldsymbol{\alpha}}(N) = \prod_{j=1}^d \exp \left( \int_0^T \log \alpha_j(s) dN_j(s) - \int_0^T \alpha_j(s)Y_j(s) ds \right) $$
*   **Estimation :** Estimation simultanée des composantes $\alpha_j(t)$ dans la même famille de fonctions.

#### C. Processus Ponctuel Spatial
*   **Contexte Vidéo :** Distribution des changements de pixels sur l'image à un instant donné ou sur un bloc spatio-temporel.
*   **Modèle d'Intensité :** $\lambda(s)$ où $s \in \mathcal{S} \subset \mathbb{R}^2$ (coordonnées spatiales).
*   **Vraisemblance (cf. `versatile_main.tex` Section Estimation of a Poisson process measure) :**
    $$ L_{\alpha}(N) = \exp \left( \int_{\mathcal{S}} \log \alpha(s) dN(s) - \int_{\mathcal{S}} \alpha(s) ds \right) $$
*   **Estimation :** Utilisation de fonctions de base 2D (ondelettes 2D, histogrammes 2D, polynômes tensoriels).

#### D. Processus Markovien des Transitions (Markovian Transition Model)
*   **Contexte Vidéo :** États discrets d'un pixel (ex: Noir, Blanc, Rouge) avec transitions permises.
*   **Modèle d'Intensité :** Intensités de transition $\alpha^{hj}(t)$ de l'état $h$ vers $j$.
*   **Vraisemblance (cf. `versatile_main.tex` Section The markovian transition model) :**
    $$ L_{\gamma}(t) = \prod_{h,j} \exp \left( \int_0^t (1 - \gamma^{hj}(s))Y^h(s)ds + \int_0^t \log \gamma^{hj}(s)dN^{hj}(s) \right) $$
    où $Y^h(s)$ est l'indicateur d'être dans l'état $h$ avant $s$.
*   **Estimation :** Estimation des fonctions $\alpha^{hj}(t)$ pour chaque paire d'états connectés.

### 1.2. Familles de Fonctions d'Intensité ($\Gamma_n$)
Conformément à `versatile_main.tex`, l'estimateur $\hat{\alpha}$ est choisi dans une suite de modèles emboîtés. Pour chaque processus ci-dessus, $\alpha$ est approximée par :
1.  **Histogrammes :** $\alpha(t) = \sum c_i \mathbb{I}_{[t_i, t_{i+1}]}(t)$.
2.  **Splines :** Splines de lissage pour capturer la continuité temporelle.
3.  **Polynômes :** Approximation globale sur $[0, T]$.
4.  **Ondelettes :** Pour capturer les variations locales brusques (adapté aux vidéos à action rapide).
5.  **Polynômes Trigonométriques :** Pour les vidéos périodiques (ex: clignotement, scènes cycliques).

### 1.3. Versatilité des Représentations
Avant d'appliquer la vraisemblance, les données sont transformées selon `versatile_pp.tex`. Pour chaque processus, on teste 4 représentations :
1.  **Time Stamps :** $\{\tau_k\}$. Précision maximale.
2.  **Counts (Histogrammes) :** $\{n_i\}$ dans des bins de temps.
3.  **Intervals :** $\{\Delta_k = \tau_{k+1} - \tau_k\}$.
4.  **Binary :** Vecteur binaire sur une grille temporelle.

Le choix de la représentation $R$ influence la forme de la vraisemblance calculée (discrète vs continue).

### 1.4. Codage Uniforme (Référence)
Conformément à `uniform_temp_coding.tex`, le codage uniforme sert de borne inférieure (baseline). Si le modèle MDL est trop complexe, on retombe sur l'uniforme.
Les longueurs de code sont :
1.  **État Complet :** $L_1 = r \log m$
2.  **Liste Sauts :** $L_2 = \log N + N \log(rm)$
3.  **Vecteur Booléen :** $L_3 = r + N \log m$
4.  **Adresse Combinatoire :** $L_4 = \log N + \log \binom{r}{N} + N \log m$

**Règle de Décision :** Le modèle MDL est validé si et seulement si :
$$ -\log L_{\hat{\alpha}} + C_n(\hat{\alpha}) < \min(L_1, L_2, L_3, L_4) $$

## 2. Cahier des Charges Informatique

### 2.1. Architecture du Système

Le système se compose de 5 modules interconnectés :

1.  **Module d'Extraction Vidéo :**
    *   Transforme la vidéo en trajectoires de pixels.
    *   Détecte les sauts (changements d'état).
    *   Segmente les données en blocs spatio-temporels.
2.  **Module de Classification de Processus :**
    *   Teste hypothèses : Est-ce spatial ? Vectoriel ? Markovien ?
    *   Sélectionne la classe de processus la plus pertinente par bloc.
3.  **Module d'Estimation d'Intensité (MDL Core) :**
    *   Pour la classe choisie, estime $\hat{\alpha}$ dans les 5 familles (Histogrammes, Splines, etc.).
    *   Calcule la pénalité de complexité $C_n(\hat{\alpha})$.
4.  **Module de Choix de Représentation :**
    *   Transforme les données dans les 4 formats (Timestamp, Count, etc.).
    *   Sélectionne le format minimisant la longueur de code pour l'intensité estimée.
5.  **Module de Codage/Décodage :**
    *   Compare le coût MDL au coût Uniforme (`uniform_temp_coding.tex`).
    *   Encode les paramètres du modèle + les résidus.
    *   Décode et reconstruit la vidéo.

### 2.2. Spécifications des Algorithmes

#### Algorithme de Sélection de Modèle (Pseudo-code)
```python
Input: VideoData, Classes = [Real, Vector, Spatial, Markov], Families = [Hist, Spline, Wavelet...], Reps = [Time, Count, Binary, Interval]
BestScore = Infinity
BestConfig = None

# 1. Calcul du bound Uniforme (Reference)
L_uniform = min( CalcL1(), CalcL2(), CalcL3(), CalcL4() ) # Cf. uniform_temp_coding.tex

# 2. Boucle sur les Classes de Processus
for ProcessClass in Classes:
    # 3. Boucle sur les Représentations
    for Rep in Reps:
        DataTransformed = Transform(VideoData, Rep)
        
        # 4. Boucle sur les Familles d'Intensité
        for Family in Families:
            # Estimation de l'intensité alpha
            alpha_hat = EstimateIntensity(DataTransformed, Family)
            
            # Calcul de la Vraisemblance (Log-Likelihood)
            # Utiliser les formules spécifiques de versatile_main.tex selon ProcessClass
            LogLikelihood = ComputeLikelihood(DataTransformed, alpha_hat, ProcessClass)
            
            # Calcul de la Complexité (Penalité)
            Complexity = ComputeComplexity(alpha_hat, Family)
            
            # Score MDL
            MDL_Score = -LogLikelihood + Complexity
            
            # Comparaison
            if MDL_Score < BestScore and MDL_Score < L_uniform:
                BestScore = MDL_Score
                BestConfig = (ProcessClass, Rep, Family, alpha_hat)

# 5. Décision Finale
if BestConfig is not None:
    Encode(BestConfig)
else:
    EncodeUniform(L_uniform)
```

#### Détails d'Implémentation des Vraisemblances
*   **Pour le Processus Markovien :** Implémenter la matrice de comptage $N^{hj}$ et le processus à risque $Y^h$. Utiliser la formule de vraisemblance produit sur $h,j$ de `versatile_main.tex`.
*   **Pour le Processus Spatial :** Discrétiser l'espace en grille. L'intégrale spatiale devient une somme sur les pixels/blocs.
*   **Pour le Processus Vectoriel :** Paralléliser l'estimation sur les composantes du vecteur.

### 2.3. Structures de Données
*   **Objet Processus :** Doit contenir `type` (enum), `intensity_params` (array), `representation_type` (enum), `family_type` (enum).
*   **Grille Temporelle :** Paramètre $r$ (nombre de bins) ajustable dynamiquement selon `Uniform_temporal.tex` (proposition sur $r_n$).
*   **Palette de Couleurs :** Paramètre $m$ (nombre d'états) issu de la quantification vidéo.

### 2.4. Exigences de Performance
*   **Convergence :** L'estimateur doit satisfaire les bornes de convergence en distance de Hellinger (Théorème principal de `versatile_main.tex`).
*   **Temps Réel :** L'estimation d'intensité (ex: histogramme) doit être $O(N)$ ou $O(r)$. Les splines/ondelettes peuvent être plus lourdes et réservées au codage "offline".
*   **Robustesse :** Si $N$ (nombre de sauts) est trop faible pour estimer une intensité complexe, le système doit automatiquement basculer vers le codage uniforme ou une famille simple (Histogramme 1 bin).

### 2.5. Livrables
1.  **Bibliothèque Mathématique :** Implémentation des vraisemblances pour les 4 classes de processus.
2.  **Moteur d'Estimation :** Solveurs pour Histogrammes, Splines, Ondelettes.
3.  **Codeur/Decodeur :** Implémentation des codes universels (`uniform_temp_coding.tex`) et arithmétiques.
4.  **Rapport de Performance :** Comparaison des taux de compression selon la classe de processus détectée.

## 3. Justification Théorique (Lien avec les Documents)

1.  **Consistance de l'Estimateur :** Le théorème principal de `versatile_main.tex` garantit que pour chaque classe de processus (Poisson, Markovien, Diffusion), l'estimateur MDL $\hat{P}_n$ converge vers la vraie distribution $P$ avec un taux lié à la distance de Hellinger :
    $$ H^2(P, \hat{P}_n) \leq \min_{Q \in \Gamma_n} \left[ H^2(P, Q) + \frac{1}{n}C_n(Q) \right] $$
    Cela valide l'approche d'estimation d'intensité pour chaque type de processus.

2.  **Optimalité du Codage Uniforme :** Les propositions de `uniform_temp_coding.tex` (Proposition 1 et 2) fournissent les bornes exactes ($L_1$ à $L_4$) qui servent de seuil de décision. Le système ne doit jamais coder plus long que $L_4$ (Adresse Combinatoire).

3.  **Versatilité :** `versatile_pp.tex` justifie le changement de représentation. Par exemple, si l'intensité est très variable, la représentation "Time Stamps" est préférée. Si l'intensité est faible et régulière, "Binary" ou "Count" peut être plus efficace pour le codage de la structure temporelle.

4.  **Modèle Multiplicatif :** `Uniform_temporal.tex` valide l'utilisation du modèle $\lambda(t) = \alpha(t)Y(t)$ (modèle d'Aalen) comme base pour tous les processus ponctuels temporels considérés (Réel, Vectoriel, Markovien).

Cette structure répond explicitement à votre exigence de prendre en compte le cadre statistique spécifique (type de processus) tout en conservant la flexibilité du codage LMD et la sécurité du codage uniforme.

### 2.2. Règles de Décision Détaillées

```python
# Pseudo-code pour l'arbre de décision

def choisir_configuration(bloc_video):
    # Étape 1 : Extraction caractéristiques
    m = compter_couleurs(bloc_video)
    N = compter_sauts(bloc_video)
    r = resolution_temporelle
    H_s = calculer_homogeneite_spatiale(bloc_video)
    lambda_avg = N / (r * nombre_pixels)
    rho_corr = calculer_correlation_spatiale(bloc_video)
    var_delta_tau = variance_intervalles_sauts(bloc_video)
    
    # Étape 2 : Classification processus
    if H_s > 0.7 and rho_corr > 0.8:
        type_processus = "SPATIAL"
        famille_intensite = "Ondelettes_2D"
    elif m > 1 and transitions_structurees(bloc_video):
        type_processus = "MARKOVIEN"
        famille_intensite = "Histogrammes"
    elif canaux_correles(bloc_video):  # R,G,B
        type_processus = "VECTORIEL"
        famille_intensite = "Splines"
    else:
        type_processus = "REEL"
        famille_intensite = "Polynomes_Trigonometriques"
    
    # Étape 3 : Décomposition monochromatique
    processus_mono = []
    for c in range(m):
        N_c = extraire_processus_couleur(bloc_video, c)
        processus_mono.append(N_c)
    
    # Étape 4 : Choix représentation
    if lambda_avg < 0.1:
        representation = "TIME_STAMPS"
    elif N / r > 0.5:
        representation = "BINARY"
    elif var_delta_tau / mean_delta_tau**2 < 0.5:
        representation = "INTERVAL"
    else:
        representation = "COUNT"
    
    # Étape 5 : Estimation intensité MDL
    meilleur_MDL = infinity
    meilleure_famille = None
    for famille in [Histogrammes, Splines, Polynomes, Ondelettes, Trigono]:
        alpha_hat = estimer_intensite(processus_mono, famille)
        vraisemblance = calculer_vraisemblance(processus_mono, alpha_hat, type_processus)
        complexite = calculer_complexite(alpha_hat, famille)
        MDL = -log(vraisemblance) + complexite
        if MDL < meilleur_MDL:
            meilleur_MDL = MDL
            meilleure_famille = famille
    
    # Étape 6 : Comparaison avec codage uniforme
    L1 = r * log(m)
    L2 = log(N) + N * log(r * m)
    L3 = r + N * log(m)
    L4 = log(N) + log(comb(r, N)) + N * log(m)
    L_uniforme = min(L1, L2, L3, L4)
    
    if meilleur_MDL < L_uniforme:
        methode_codage = "MDL"
        L_final = meilleur_MDL
    else:
        methode_codage = "UNIFORME"
        L_final = L_uniforme
    
    return {
        'type_processus': type_processus,
        'representation': representation,
        'famille_intensite': meilleure_famille,
        'methode_codage': methode_codage,
        'longueur_code': L_final
    }
```

### 3.3. Formules de Calcul des Caractéristiques

#### Homogénéité Spatiale (H_s)
$$H_s = 1 - \frac{\sigma_{couleurs}}{m_{max}}$$
où $\sigma_{couleurs}$ est l'écart-type des couleurs dans le bloc et $m_{max}$ le nombre maximum de couleurs possibles.

#### Corrélation Spatiale (ρ_corr)
$$\rho_{corr} = \frac{1}{|V|} \sum_{(i,j) \in V} \frac{Cov(C_i, C_j)}{\sigma_{C_i} \sigma_{C_j}}$$
où $V$ est l'ensemble des paires de pixels voisins.

#### Régularité Temporelle
$$R_{temp} = \frac{E[\Delta \tau]^2}{Var[\Delta \tau]}$$
- $R_{temp} > 1$ : Processus régulier → Représentation INTERVAL
- $R_{temp} \approx 1$ : Processus Poisson → Représentation TIME_STAMPS
- $R_{temp} < 1$ : Processus groupé → Représentation BINARY

## 1. Résumé Exécutif

Ce projet vise à développer un framework de compression vidéo innovant basé sur la modélisation des trajectoires de pixels comme des processus ponctuels. Contrairement aux méthodes standards (DCT, MPEG), cette approche exploite l'historique temporel individuel des pixels. Le système utilise le principe de la **Longueur Minimale de Description (MDL)** pour sélectionner automatiquement le modèle statistique optimal parmi plusieurs classes de processus (Réel, Vectoriel, Spatial, Markovien) et plusieurs représentations (Temporelle, Comptage, Intervalle, Binaire). Un codage uniforme sert de borne de référence pour garantir la robustesse. Ce rapport formalise le modèle mathématique, l'architecture logicielle, les heuristiques de décision et le protocole d'entraînement.

---

## 2. Cadre Théorique

### 2.1. Modèle Général des Processus Ponctuels
Conformément à `Uniform_temporal.tex` et `versatile_main.tex`, un processus ponctuel marqué $N$ est observé sur un intervalle $[0, T]$. Un réalisé est noté $\omega = \{(\tau_k, m_k)\}_{k=1}^N$, où $\tau_k$ sont les temps de sauts et $m_k \in \mathcal{E}$ les marques (couleurs).

Le modèle d'intensité stochastique utilisé est le **modèle multiplicatif d'Aalen** :
$$ \lambda(t) = \alpha(t)Y(t) $$
où $\alpha(t)$ est l'intensité de base (inconnue) et $Y(t)$ un processus prévisible observable.

### 2.2. Classes de Processus et Vraisemblances
Le système doit être capable de modéliser quatre classes principales, dont les vraisemblances sont définies dans `versatile_main.tex` :

1.  **Processus Ponctuel Réel (Standard) :**
    *   Intensité scalaire $\lambda(t)$.
    *   Vraisemblance : $L_{\alpha}(N) = \exp \left( \int_0^T \log(\alpha(s)) dN(s) - \int_0^T \alpha(s)Y(s) ds \right)$.
2.  **Processus Ponctuel Vectoriel (Multivarié) :**
    *   Vecteur d'intensités $\boldsymbol{\lambda}(t) = (\lambda_1(t), \dots, \lambda_d(t))$.
    *   Vraisemblance : Produit des vraisemblances marginales (ou copules).
3.  **Processus Ponctuel Spatial :**
    *   Intensité $\lambda(s)$ sur un domaine spatial $\mathcal{S}$.
    *   Vraisemblance : $L_{\alpha}(N) = \exp \left( \int_{\mathcal{S}} \log \alpha(s) dN(s) - \int_{\mathcal{S}} \alpha(s) ds \right)$.
4.  **Processus Markovien des Transitions :**
    *   Intensités de transition $\alpha^{hj}(t)$ entre états $h$ et $j$.
    *   Vraisemblance (cf. `versatile_main.tex` Section 4.4) :
        $$ L_{\gamma}(t) = \prod_{h,j} \exp \left( \int_0^t (1 - \gamma^{hj}(s))Y^h(s)ds + \int_0^t \log \gamma^{hj}(s)dN^{hj}(s) \right) $$

### 2.3. Principe MDL (Minimum Description Length)
Conformément à `versatile_main.tex`, l'estimateur $\hat{P}_n$ est choisi pour minimiser la complexité stochastique :
$$ \hat{P}_n = \arg \min_{Q \in \mathcal{P}} \left( -\log Q(X^n) + C_n(Q) \right) $$
où $-\log Q$ est la longueur de code des données et $C_n(Q)$ la pénalité de complexité du modèle.
Le théorème principal garantit la convergence en distance de Hellinger :
$$ H^2(P, \hat{P}_n) \leq \min_{Q \in \Gamma_n} \left[ H^2(P, Q) + \frac{1}{n} C_n(Q) \right] $$

### 2.4. Codage Uniforme (Référence)
Conformément à `uniform_temp_coding.tex`, le codage uniforme sert de borne inférieure (baseline). Si le modèle MDL est trop complexe, on retombe sur l'uniforme. Les longueurs de code sont :
1.  **État Complet :** $L_1 = r \log m$
2.  **Liste Sauts :** $L_2 = \log N + N \log(rm)$
3.  **Vecteur Booléen :** $L_3 = r + N \log m$
4.  **Adresse Combinatoire :** $L_4 = \log N + \log \binom{r}{N} + N \log m$

**Règle de Décision :** Le modèle MDL est validé si et seulement si :
$$ -\log L_{\hat{\alpha}} + C_n(\hat{\alpha}) < \min(L_1, L_2, L_3, L_4) $$

## 3. Méthodologie de Codage Vidéo

### 3.1. Extraction et Transformation
Le module d'extraction transforme le flux vidéo en processus ponctuels :
1.  **Quantification Colorimétrique :** Réduction de l'espace colorimétrique à $m$ couleurs dominantes.
2.  **Détection de Sauts :** Identification des instants $t$ où $C(x,y,t) \neq C(x,y,t-1)$ pour chaque pixel.
3.  **Décomposition Monochromatique :** Au lieu de coder un processus marqué, nous définissons $m$ processus ponctuels simples $N_i^{(c)}$ par source spatiale.
    *   *Avantage :* Élimine le terme de codage explicite des marques ($N \log m$). La couleur est portée par l'indice du processus.

### 3.2. Versatilité des Représentations
Conformément à `versatile_pp.tex`, pour chaque processus, on teste 4 représentations :
1.  **Time Stamps :** Liste exacte des temps $\{\tau_k\}$.
2.  **Counts (Histogrammes) :** Vecteur de comptes dans des bins de temps fixes.
3.  **Intervals :** Durées entre sauts $\{\Delta_k\}$.
4.  **Binary :** Vecteur de bits indiquant la présence d'un saut dans chaque bin.

### 3.3. Estimation d'Intensité
L'intensité $\alpha(t)$ est estimée dans plusieurs familles de fonctions ($\Gamma_n$) :
*   Histogrammes (fonctions en escalier).
*   Splines (lissage temporel).
*   Polynômes (tendance globale).
*   Ondelettes (variations locales).
*   Polynômes Trigonométriques (périodicité).

## 4. Arbre de Décision et Heuristiques

### 4.1. Caractéristiques à Extraire
Pour chaque bloc vidéo, les caractéristiques suivantes sont calculées pour alimenter l'arbre de décision :
*   $m$ : Nombre de couleurs distinctes.
*   $N$ : Nombre total de sauts.
*   $r$ : Nombre de bins temporels.
*   $H_s$ : Homogénéité spatiale (variance inter-pixels).
*   $\lambda_{avg}$ : Intensité moyenne des sauts ($N/T$).
*   $\rho_{corr}$ : Corrélation spatiale entre pixels voisins.
*   $R_{temp}$ : Régularité temporelle ($E[\Delta \tau]^2 / Var[\Delta \tau]$).

### 4.2. Logique de Décision (Synthèse)
1.  **Classification du Processus :**
    *   Si $H_s > 0.7$ et $\rho_{corr} > 0.8$ $\rightarrow$ **Spatial**.
    *   Si $m > 1$ et transitions structurées $\rightarrow$ **Markovien**.
    *   Si canaux corrélés (R,G,B) $\rightarrow$ **Vectoriel**.
    *   Sinon $\rightarrow$ **Réel**.
2.  **Choix de la Représentation :**
    *   Si $\lambda_{avg} < 0.1$ $\rightarrow$ **Time Stamps**.
    *   Si $N/r > 0.5$ $\rightarrow$ **Binary**.
    *   Si $R_{temp} > 1$ (régulier) $\rightarrow$ **Interval**.
    *   Sinon $\rightarrow$ **Count**.
3.  **Sélection MDL vs Uniforme :**
    *   Calculer $L_{MDL}$ pour la meilleure famille d'intensité.
    *   Calculer $L_{uniforme} = \min(L_1, L_2, L_3, L_4)$.
    *   Choisir le minimum.

## 5. Architecture Logicielle

Le système se compose de 5 modules interconnectés :

1.  **Module d'Extraction Vidéo :**
    *   Transforme la vidéo en trajectoires de pixels.
    *   Détecte les sauts et segmente les données en blocs spatio-temporels.
2.  **Module de Classification de Processus :**
    *   Teste les hypothèses (Spatial, Vectoriel, Markovien, Réel).
    *   Sélectionne la classe de processus la plus pertinente par bloc.
3.  **Module d'Estimation d'Intensité (MDL Core) :**
    *   Estime $\hat{\alpha}$ dans les 5 familles (Histogrammes, Splines, etc.).
    *   Calcule la pénalité de complexité $C_n(\hat{\alpha})$.
4.  **Module de Choix de Représentation :**
    *   Transforme les données dans les 4 formats (Timestamp, Count, etc.).
    *   Sélectionne le format minimisant la longueur de code.
5.  **Module de Codage/Décodage :**
    *   Compare le coût MDL au coût Uniforme.
    *   Encode les paramètres du modèle + les résidus (codage arithmétique/universel).
    *   Décode et reconstruit la vidéo.

## 2. Cadre Mathématique et Classes de Processus

Conformément à `Uniform_temporal.tex` et `versatile_main.tex`, le système doit gérer deux approches distinctes pour la modélisation des états (couleurs).

### 2.1. Processus Ponctuel Réel Marqué vs. Somme de Processus Monochromatiques

Pour un pixel donné, l'historique des couleurs peut être modélisé de deux façons :

1.  **Processus Ponctuel Marqué (Standard) :**
    *   Un seul processus $N$ avec des sauts $\tau_k$ et des marques $m_k \in \{1, \dots, m\}$.
    *   **Vraisemblance :** Inclut un terme de codage des marques ($N \log m$).
    *   **Référence :** `Uniform_temporal.tex` Eq. (2).
    *   **Intensité :** $\lambda(t, m) = \alpha(t, m)Y(t)$.

2.  **Somme de Processus Monochromatiques (Optimisation) :**
    *   Décomposition en $m$ processus simples $N^{(c)}$ pour chaque couleur $c$.
    *   **Avantage :** La couleur est implicite dans l'indice du processus. Élimine le terme $N \log m$ du codage des marques.
    *   **Vraisemblance :** Produit des vraisemblances marginales pour chaque couleur.
    *   **Référence :** `uniform_temp_coding.tex` (Comparaison entre codage avec et sans marques).

### 2.2. Processus Ponctuel Vectoriel (Corrélations)

Pour des blocs de pixels ou des canaux (R, G, B) :

1.  **Approche Marginale (Indépendante) :**
    *   Chaque dimension est codée comme un processus scalaire indépendant.
    *   **Structure :** Somme de processus monochromatiques par dimension.
2.  **Approche Conjointe (Vectorielle) :**
    *   Prend en compte les corrélations entre états/dimensions.
    *   **Modèle :** Processus de Markov ou Copules.
    *   **Vraisemblance :** Inclut les termes de transition ou de dépendance.
    *   **Référence :** `versatile_main.tex` Section 4.4 (Markovian transition model).

## 3. Structures de Données et Pré-Codage (Représentation)

Le choix de la représentation précède le codage effectif (étape de "pré-codage"). Chaque approche (Marquée, Monochromatique, Vectorielle) génère une structure de données spécifique. Conformément à `versatile_pp.tex` et `uniform_temp_coding.tex`, quatre représentations principales sont supportées.

### 3.1. Les 4 Représentations Canoniques

| ID | Nom | Description | Structure de Données | Référence |
|----|-----|-------------|----------------------|-----------|
| **R1** | **Time Stamps** | Liste exacte des temps de sauts. | `List<Integer>` (Dates) + `List<Integer>` (Couleurs si marqué) | `versatile_pp.tex` |
| **R2** | **Count (Histogramme)** | Nombre de sauts par bin temporel. | `Array<int>` (Comptes par bin) | `versatile_pp.tex` |
| **R3** | **Boolean Vector** | Présence (1) ou absence (0) de saut par bin. | `BitSet` ou `Array<bool>` de taille $r$ | `uniform_temp_coding.tex` ($L_3$) |
| **R4** | **Combinatorial Address** | Dérivé du Booléen. Index de la configuration. | `Integer N` (Nb sauts) + `Integer Index` (Parmi $\binom{r}{N}$) | `uniform_temp_coding.tex` ($L_4$) |

*Note sur R4 :* Comme précisé dans la demande, à partir du vecteur booléen, on construit la représentation $(N, \text{Index})$. Cela correspond à la longueur de code $L_4 = \log N + \log \binom{r}{N} + N \log m$ (avec $m=1$ pour monochromatique).

### 3.2. Primitives d'Accès (API de Pré-Codage)

Pour chaque type de codage, des primitives d'accès aux couleurs et aux dates doivent être implémentées :

*   **`get_jump_times()` :** Retourne la liste des $\tau_k$. (Optimisé pour R1, R4).
*   **`get_color_at(t)` :** Retourne la couleur au temps $t$. (Optimisé pour R2, R3).
*   **`get_count_in_bin(b)` :** Retourne le nombre de sauts dans le bin $b$. (Optimisé pour R2).
*   **`has_jump_at(t)` :** Retourne booléen. (Optimisé pour R3).

## 4. Stratégies de Compression Effectives

Une fois la représentation choisie, la compression effective suit l'une des trois méthodes ci-dessous, classées par complexité croissante.

### 4.1. Codage Universel sur les Entiers
*   **Usage :** Lorsque les paramètres (dates, comptes) ne sont pas bornés ou pour éviter de coder la borne maximale.
*   **Méthode :** Codes de Elias ($\gamma, \delta$) ou Rissanen.
*   **Référence :** `uniform_temp_coding.tex` Section 2.2 (Universal encoding of integers).
*   **Formule :** $L(n) \approx \log n + 2 \log \log n$.
*   **Avantage :** Robustesse, pas besoin de connaître $r$ ou $m$ à l'avance.

### 4.2. Codage par Loi Uniforme (Baseline)
*   **Usage :** Référence minimale (borne inférieure). Utilisé si l'estimation d'intensité échoue ou est trop coûteuse.
*   **Méthode :** Supposition que les sauts et les couleurs sont uniformément distribués.
*   **Longueurs de Code (`uniform_temp_coding.tex` Prop. 1) :**
    1.  $L_1 = r \log m$ (État complet)
    2.  $L_2 = \log N + N \log(rm)$ (Liste sauts)
    3.  $L_3 = r + N \log m$ (Booléen)
    4.  $L_4 = \log N + \log \binom{r}{N} + N \log m$ (Combinatoire)
*   **Décision :** Le système choisit $\min(L_1, L_2, L_3, L_4)$ selon le nombre de sauts $N$ observé.

### 4.3. Codage Statistique MDL (3 Étapes)
*   **Usage :** Compression optimale lorsque des régularités temporelles existent.
*   **Référence :** `versatile_main.tex` (MDL estimator).
*   **Étape 1 : Sélection du Modèle Statistique**
    *   Choix de la classe de processus (Poisson, Markov, etc.) et de la famille d'intensité (Histogramme, Splines, Ondelettes).
    *   Critère : Minimisation de $-\log L + C_n$.
*   **Étape 2 : Estimation de la Fonction d'Intensité**
    *   Calcul de $\hat{\alpha}(t)$ dans le modèle multiplicatif $\lambda(t) = \alpha(t)Y(t)$.
    *   Référence : `Uniform_temporal.tex` (Aalen model).
*   **Étape 3 : Codage avec l'Intensité Cumulée**
    *   Utilisation de la fonction de répartition cumulative pour le codage arithmétique.
    *   La probabilité d'un saut dans un intervalle est donnée par $\int \hat{\alpha}(t) dt$.
    *   Référence : `versatile_main.tex` (Likelihood coding).

## 6. Implémentation et Complexité

### 6.1. Gestion des Structures de Données
*   **Monochromatique :** Tableau de listes de sauts `List<Integer>[] jumps_by_color`.
*   **Vectoriel :** Tableau d'objets Processus `Process[] dimensions`.
*   **Combinatoire (R4) :** Nécessite une fonction efficace pour calculer l'index combinatoire $\sum \binom{r-i}{N-k}$ sans overflow (utilisation de logarithmes ou grandes entier).

### 6.2. Primitives de Codage
*   **`encode_universal(int n)` :** Implémente le code $\delta$ de Rissanen (`uniform_temp_coding.tex`).
*   **`encode_uniform(int N, int r, int m)` :** Calcule et encode selon la formule $L_i$ choisie.
*   **`encode_statistical(List<Integer> jumps, Function intensity)` :** Utilise l'intensité cumulée pour mapper les sauts dans $[0,1]$ puis code arithmétique.

### 6.3. Complexité Algorithmique
*   **Estimation Intensité :** $O(N)$ pour histogrammes, $O(N \log N)$ pour Splines/Ondelettes.
*   **Calcul Index Combinatoire :** $O(N)$ ou $O(r)$ selon implémentation.
*   **Décision MDL :** Doit rester inférieure au gain de compression espéré.

## 2. Modèle Mathématique et Termes de Couleur

Conformément à `uniform_temp_coding.tex` et `Uniform_temporal.tex`, la longueur de code dépend de la stratégie de gestion des marques (couleurs/états).

### 2.1. Codage par Loi Uniforme et Universel (Processus Marqué)
Dans le cas standard d'un processus ponctuel marqué $N = \{(\tau_k, m_k)\}$, les formules de `uniform_temp_coding.tex` (Proposition 1) incluent un terme de codage de la marque pour **chaque saut** :

*   $L_2 = \log N + N \log r + \mathbf{N \log m}$ (Liste sauts + couleurs)
*   $L_3 = r + \mathbf{N \log m}$ (Booléen + couleurs)
*   $L_4 = \log N + \log C_r^N + \mathbf{N \log m}$ (Combinatoire + couleurs)

Ici, le terme $N \log m$ représente le coût de codage de la couleur à chaque changement d'état.

### 2.2. Codage État par État (Décomposition Monochromatique)
Conformément à l'approche de décomposition discutée, le processus est vu comme une somme de $m$ processus monochromatiques $N^{(c)}$.
*   **Avantage :** Les sauts dans $N^{(c)}$ n'ont pas de marque. Le terme $N \log m$ disparaît des formules de codage temporel.
*   **Coût de Pré-codage :** Il faut encoder la correspondance "Indice du Processus $\leftrightarrow$ Couleur".
    *   Si la palette est globale : Coût nul (implicite).
    *   Si la palette est locale (par bloc) : Coût $\approx m \times \log |\mathcal{E}_{global}|$ ou $\log m$ par processus actif.
*   **Nouvelle Formule (exemplaire) :**
    $$ L_{mono} = \sum_{c=1}^m L_{time}(N^{(c)}) + \mathbf{C_{color\_map}} $$
    Où $C_{color\_map}$ est codé **une fois par bloc** (et non par saut).

### 2.3. Distinction au Niveau du Pré-Codage
Le module de pré-codage doit sélectionner le **Mode de Couleur** :
*   **Mode M (Marked) :** Couleur codée par saut. Terme $N \log m$ inclus.
*   **Mode S (State-by-State) :** Couleur codée par bloc/processus. Terme $N \log m$ supprimé, remplacé par $C_{color\_map}$.

---

## 3. Identification Unique des Méthodes (Method ID)

Pour assurer la réversibilité du codage, chaque bloc compressé doit être précédé d'un en-tête identifiant la stratégie utilisée. Nous définissons un **Code Méthode (Method ID)** binaire concaténé.

### 3.1. Structure du Code Méthode
Le Code Méthode est composé de 4 champs :

| Champ | Bits | Description | Valeurs Exemples |
|-------|------|-------------|------------------|
| **PROC** | 2 | Classe de Processus | `00`: Réel, `01`: Vectoriel, `10`: Spatial, `11`: Markovien |
| **REP** | 2 | Représentation (`versatile_pp.tex`) | `00`: Timestamp, `01`: Count, `10`: Interval, `11`: Boolean/Comb |
| **COMP** | 2 | Méthode Compression | `00`: Uniforme, `01`: Universel Entier, `10`: MDL Statistique, `11`: Réservé |
| **COL** | 1 | Mode Couleur | `0`: Marqué (Terme $N \log m$), `1 État par État (Terme $C_{map}$) |

**Exemple :** `0110011`
*   `01` (Vectoriel)
*   `10` (Interval)
*   `01` (Universel Entier)
*   `1` (État par État)

### 3.2. Intégration dans le Flux (Bitstream)
Chaque bloc de données est structuré ainsi :
```
[ Header Bloc ] [ Code Méthode (7 bits) ] [ Paramètres Modèle ] [ Données Codées ] [ Checksum ]
```
*   **Header Bloc :** Coordonnées spatiales, taille, frame de début/fin.
*   **Paramètres Modèle :** Dépend de la méthode (ex: coefficients de spline pour MDL, ou palette pour Mode Couleur État par État).

## 4. Processus d'Entraînement et Génération d'Heuristiques

Pour éviter un codage exhaustif en temps réel (trop coûteux), un processus d'apprentissage hors-ligne est conçu pour prédire le meilleur **Code Méthode** à partir des caractéristiques du bloc.

### 4.1. Phase 1 : Codage Exhaustif (Offline)
Sur un dataset d'entraînement (vidéos variées), chaque bloc est codé avec **toutes les combinaisons possibles** de méthodes.

*   **Entrée :** Bloc Vidéo $B$.
*   **Boucle :** Pour chaque `Code Méthode` $k \in \{1, \dots, K\}$ :
    1.  Appliquer le pré-codage (extraction sauts, décomposition couleur).
    2.  Estimer les paramètres (si MDL).
    3.  Calculer la longueur de code totale $L_k(B)$.
*   **Sélection :** Identifier la méthode optimale $k^* = \arg \min_k L_k(B)$.
*   **Enregistrement :** Stocker le tuple $(Features(B), k^*, \{L_k\})$.

### 4.2. Phase 2 : Extraction de Caractéristiques (Features)
Pour chaque bloc, les caractéristiques suivantes sont calculées (cf. `uniform_temp_coding.tex` et `versatile_main.tex`) :

1.  **Statistiques Temporelles :**
    *   $N$ : Nombre total de sauts.
    *   $r$ : Nombre de bins temporels.
    *   $\lambda_{avg} = N/r$ : Densité de sauts.
    *   $Var(\Delta \tau)$ : Régularité des intervalles.
2.  **Statistiques de Couleur :**
    *   $m$ : Nombre de couleurs distinctes dans le bloc.
    *   $Entropy(C)$ : Entropie spatiale des couleurs.
    *   $Cost_{marked} = N \log m$.
    *   $Cost_{state} = C_{color\_map}$.
3.  **Statistiques Spatiales :**
    *   $H_{spatiale}$ : Homogénéité du bloc.
    *   $\rho_{corr}$ : Corrélation avec les blocs voisins.

### 4.3. Phase 3 : Apprentissage du Modèle de Décision
Un modèle de classification est entraîné sur les données collectées.

*   **Input :** Vecteur de caractéristiques $X = [N, r, m, \lambda_{avg}, \dots]$.
*   **Output :** Classe $Y = k^*$ (Le meilleur Code Méthode).
*   **Algorithmes Suggérés :**
    *   **Arbre de Décision (CART) :** Pour obtenir des règles interprétables (seuils sur $N, m$).
    *   **Random Forest / Gradient Boosting :** Pour une meilleure précision prédictive.
*   **Fonction de Perte :** Minimiser la différence entre la longueur de code prédite et la longueur de code optimale réelle (Redondance).

### 4.4. Phase 4 : Outil d'Aide à la Décision (Online)
Lors du codage réel (production) :
1.  Calculer les caractéristiques $X$ du bloc.
2.  Interroger le modèle entraîné pour obtenir le **Code Méthode Prédit** $\hat{k}$.
3.  Appliquer uniquement la méthode $\hat{k}$.
4.  (Optionnel) Vérifier rapidement si une méthode alternative simple (ex: Uniforme) est meilleure (sécurité).

## 1. Contexte et Objectifs du Projet

### 1.1. Contexte
Les techniques de compression vidéo standards (DCT, MPEG, HEVC) reposent sur la compression spatiale à intervalles de temps réguliers, sans considérer l'historique individuel des pixels. Ce projet propose une approche alternative basée sur la modélisation des trajectoires temporelles de pixels comme des **processus ponctuels**, exploitant le principe de la **Longueur Minimale de Description (MDL)** pour une compression adaptative et statistiquement optimale.

### 1.2. Objectifs Principaux
1.  **Développer un codeur/décodeur vidéo** basé sur la théorie des processus ponctuels.
2.  **Implémenter la versatilité des représentations** (Timestamp, Count, Interval, Binary/Combinatorial) conformément à `versatile_pp.tex`.
3.  **Intégrer le codage MDL** avec estimation d'intensité pour les régularités temporelles (`versatile_main.tex`).
4.  **Garantir la robustesse** via le codage uniforme de référence (`uniform_temp_coding.tex`).
5.  **Créer un système d'apprentissage d'heuristiques** pour la sélection automatique de méthode.

### 1.3. Périmètre
*   **Entrée :** Flux vidéo brut (séquences d'images).
*   **Sortie :** Flux binaire compressé (bitstream) et vidéo reconstruite.
*   **Cible :** Vidéos à faible mouvement, surveillance, drones (UAS), données spatiotemporelles massives.

```python
    donnees_training = []
    for bloc in dataset_video:
        features = extraire_features(bloc) # N, m, r, H_s, rho_corr, etc.
        meilleurs_resultats = {}
        
        # Tester toutes les combinaisons (PROC x REP x COMP x COL)
        for methode in toutes_les_methodes():
            longueur = coder_bloc(bloc, methode)
            meilleurs_resultats[methode] = longueur
        
        # Sélectionner la méthode optimale
        methode_optimale = argmin(meilleurs_resultats)
        
        # Enregistrer pour l'apprentissage
        donnees_training.append({
            'features': features,
            'label': methode_optimale,
            'gain': calculer_gain(methode_optimale, uniforme)
        })
    
    # Entraîner le modèle de décision (Random Forest / Gradient Boosting)
    modele = entrainer_modele(donnees_training)
    return modele
```

#### Algorithme de Codage (Production)
```python
def coder_bloc_production(bloc, modele_entraîne):
    """
    Phase Online : Utilisation des heuristiques pour éviter le codage exhaustif.
    """
    # 1. Extraction Features
    features = extraire_features(bloc)
    
    # 2. Prédiction Méthode via Modèle
    code_methode = modele_entraîne.predict(features)
    
    # 3. Vérification de Sécurité (Optionnel)
    L_pred = estimer_longueur(code_methode)
    L_uniforme = calculer_L_uniforme(bloc) # L1, L2, L3, L4
    if L_pred > L_uniforme:
        code_methode = set_uniforme_optimal(bloc)
    
    # 4. Encodage Effectif
    bitstream = encoder(bloc, code_methode)
    
    return bitstream
```

### 3.4. Estimation d'Intensité (MDL)
Pour la méthode **MDL Statistique**, le système doit implémenter les familles de fonctions suivantes (`versatile_main.tex`) :
1.  **Histogrammes :** $\alpha(t) = \sum c_i \mathbb{I}_{[t_i, t_{i+1}]}(t)$.
2.  **Splines :** Fonctions polynomiales par morceaux lisses.
3.  **Polynômes :** $\alpha(t) = \sum_{k=0}^d a_k t^k$.
4.  **Ondelettes :** Décomposition multi-résolution.
5.  **Polynômes Trigonométriques :** Pour données périodiques.

**Critère de Sélection :**
$$ \hat{\alpha} = \arg \min_{\alpha \in \Gamma} \left( -\log L_{\alpha}(Data) + C_n(\alpha) \right) $$

## 4. Processus d'Entraînement des Heuristiques

### 4.1. Dataset d'Entraînement
*   **Contenu :** Vidéos variées (faible mouvement, fort mouvement, animation, réel).
*   **Découpage :** Blocs spatiaux (ex: 16x16 pixels) sur des GOP (Group of Pictures).
*   **Taille Minimale :** 100 heures de vidéo annotées.

### 4.2. Caractéristiques à Extraire (Features)
| Catégorie | Paramètre | Symbole | Description |
|-----------|-----------|---------|-------------|
| **Temporel** | Nombre de sauts | $N$ | Total des changements d'état. |
| | Densité de sauts | $N/r$ | Ratio sauts / bins temporels. |
| | Régularité | $R_{temp}$ | $E[\Delta \tau]^2 / Var[\Delta \tau]$. |
| **Couleur** | Nombre de couleurs | $m$ | Couleurs distinctes dans le bloc. |
| | Coût Marqué | $C_M$ | $N \log m$. |
| | Coût État | $C_S$ | $C_{color\_map}$. |
| **Spatial** | Homogénéité | $H_s$ | Variance inter-pixels. |
| | Corrélation | $\rho_{corr}$ | Corrélation voisins. |

### 4.3. Modèle de Décision
*   **Type :** Forêt Aléatoire (Random Forest) ou Gradient Boosting.
*   **Input :** Vecteur de features.
*   **Output :** Code Méthode (Classe multi-label).
*   **Métrique d'Optimisation :** Maximiser le gain de compression moyen.
*   **Contrainte :** PSNR > 30 dB.

## 5. Métriques de Performance et Validation

### 5.1. Taux de Compression
*   **Définition :** Ratio = Taille_brute / Taille_codée.
*   **Cibles :**
    *   Vidéos faiblement dynamiques : > 2.0
    *   Vidéos fortement dynamiques : > 1.2
*   **Comparaison :** HEVC/H.265 en mode "All Intra".

### 5.2. Qualité de Reconstruction
*   **PSNR (Peak Signal-to-Noise Ratio) :** Cible > 30 dB.
*   **SSIM (Structural Similarity) :** Cible > 0.95.
*   **Distance de Hellinger :** $H^2(P_{original}, P_{estimée}) < 0.1$ (`versatile_main.tex`).

### 5.3. Performance Computationnelle
*   **Temps d'Estimation :** < 10ms par bloc (pour temps réel).
*   **Mémoire :** < 100 MB pour le modèle d'heuristiques.
*   **Complexité :** $O(n \cdot r \cdot m)$ où $n$=pixels, $r$=bins, $m$=couleurs.

### 5.4. Robustesse
*   **Généralisation :** Performance sur vidéos non vues pendant l'entraînement.
*   **Fallback :** Le système doit toujours pouvoir retomber sur le codage uniforme ($L_4$) si MDL échoue.

### 1.3. Règles de Gestion

1. **RG1** : Un bloc vidéo est associé à un et un seul type de processus optimal.
2. **RG2** : Chaque processus utilise exactement une représentation.
3. **RG3** : Chaque représentation utilise exactement une méthode de compression.
4. **RG4** : La méthode MDL nécessite une famille de fonctions d'intensité.
5. **RG5** : Le mode couleur est choisi au niveau du bloc (Marqué ou État par État).
6. **RG6** : Le code méthode (7 bits) identifie de manière unique la combinaison (PROC, REP, COMP, COL).
7. **RG7** : Les données d'entraînement sont collectées par codage exhaustif hors-ligne.

### 2.2. Implémentation des Classes Principales

#### 2.2.1. Classes de Processus Ponctuels (`core/process.py`)

```python
"""
core/process.py - Classes de processus ponctuels
Référence: versatile_main.tex Section 4, Uniform_temporal.tex Section 2
"""

from abc import ABC, abstractmethod
from typing import List, Tuple, Optional
import numpy as np
from enum import Enum

class ProcessType(Enum):
    REAL = 0
    VECTOR = 1
    SPATIAL = 2
    MARKOV = 3

class PointProcess(ABC):
    """Classe de base pour les processus ponctuels"""
    
    def __init__(self, time_interval: Tuple[float, float], process_id: str = None):
        self.time_interval = time_interval
        self.process_id = process_id
        self.jump_times: List[float] = []
        self.n_events = 0
    
    @abstractmethod
    def likelihood(self, intensity_func) -> float:
        """Calcule la vraisemblance du processus"""
        pass
    
    @abstractmethod
    def get_intensity(self) -> callable:
        """Retourne la fonction d'intensité"""
        pass
    
    def add_jump(self, time: float):
        """Ajoute un saut au processus"""
        if self.time_interval[0] <= time <= self.time_interval[1]:
            self.jump_times.append(time)
            self.n_events += 1
    
    def get_jump_times(self) -> List[float]:
        """Primitive d'accès aux temps de sauts"""
        return sorted(self.jump_times)

class RealMarkedProcess(PointProcess):
    """Processus ponctuel réel avec marques (couleurs)"""
    
    def __init__(self, time_interval: Tuple[float, float], n_states: int, **kwargs):
        super().__init__(time_interval, **kwargs)
        self.process_type = ProcessType.REAL
        self.n_states = n_states
        self.marks: List[int] = []  # Couleurs associées aux sauts
    
    def add_jump(self, time: float, mark: int):
        """Ajoute un saut avec marque"""
        super().add_jump(time)
        self.marks.append(mark)
    
    def likelihood(self, intensity_func) -> float:
        """
        Vraisemblance pour processus marqué
        Référence: Uniform_temporal.tex Eq. (2)
        """
        log_likelihood = 0.0
        for tau, m in zip(self.jump_times, self.marks):
            log_likelihood += np.log(intensity_func(tau, m))
        
        # Terme d'intégrale
        integral = 0.0
        for c in range(self.n_states):
            integral += self._numerical_integral(
                lambda t: intensity_func(t, c), 
                self.time_interval[0], 
                self.time_interval[1]
            )
        
        return np.exp(log_likelihood - integral)
    
    def _numerical_integral(self, f, a, b, n=1000):
        """Intégration numérique simple"""
        x = np.linspace(a, b, n)
        return np.trapz(f(x), x)

class MonochromaticProcess(PointProcess):
    """Processus monochromatique (décomposition par couleur)"""
    
    def __init__(self, time_interval: Tuple[float, float], color_id: int, **kwargs):
        super().__init__(time_interval, **kwargs)
        self.color_id = color_id
        self.process_type = ProcessType.REAL
    
    def likelihood(self, intensity_func) -> float:
        """
        Vraisemblance pour processus monochromatique
        Référence: versatile_main.tex Section 4.2
        """
        log_likelihood = sum(np.log(intensity_func(t)) for t in self.jump_times)
        integral = self._numerical_integral(
            intensity_func, 
            self.time_interval[0], 
            self.time_interval[1]
        )
        return np.exp(log_likelihood - integral)

class VectorProcess(PointProcess):
    """Processus ponctuel vectoriel (canaux corrélés)"""
    
    def __init__(self, time_interval: Tuple[float, float], dimensions: int, **kwargs):
        super().__init__(time_interval, **kwargs)
        self.process_type = ProcessType.VECTOR
        self.dimensions = dimensions
        self.component_processes: List[PointProcess] = []
        self.correlations: Optional[np.ndarray] = None
    
    def add_component(self, process: PointProcess):
        """Ajoute une composante au processus vectoriel"""
        self.component_processes.append(process)
    
    def likelihood(self, intensity_funcs: List[callable]) -> float:
        """
        Vraisemblance pour processus vectoriel
        Référence: versatile_main.tex Section 4
        """
        if len(intensity_funcs) != self.dimensions:
            raise ValueError("Nombre de fonctions d'intensité incorrect")
        
        # Produit des vraisemblances marginales (indépendance)
        likelihood = 1.0
        for proc, intensity in zip(self.component_processes, intensity_funcs):
            likelihood *= proc.likelihood(intensity)
        
        # TODO: Ajouter terme de corrélation si modèle conjoint
        if self.correlations is not None:
            likelihood *= self._correlation_term()
        
        return likelihood
    
    def _correlation_term(self) -> float:
        """Terme de corrélation entre composantes"""
        # Implémentation dépend du modèle de copule choisi
        return 1.0

class MarkovTransitionProcess(PointProcess):
    """Processus markovien des transitions"""
    
    def __init__(self, time_interval: Tuple[float, float], n_states: int, **kwargs):
        super().__init__(time_interval, **kwargs)
        self.process_type = ProcessType.MARKOV
        self.n_states = n_states
        self.transition_counts: np.ndarray = np.zeros((n_states, n_states))
        self.state_occupation: np.ndarray = np.zeros(n_states)
    
    def add_transition(self, from_state: int, to_state: int, time: float):
        """Ajoute une transition entre états"""
        super().add_jump(time)
        self.transition_counts[from_state, to_state] += 1
    
    def likelihood(self, intensity_matrix: np.ndarray) -> float:
        """
        Vraisemblance pour processus markovien
        Référence: versatile_main.tex Section 4.4 Eq. (4.4)
        """
        log_likelihood = 0.0
        
        for h in range(self.n_states):
            for j in range(self.n_states):
                if h != j:
                    # Terme d'intégrale
                    log_likelihood += (1 - intensity_matrix[h, j]) * self.state_occupation[h]
                    # Terme de sauts
                    if self.transition_counts[h, j] > 0:
                        log_likelihood += self.transition_counts[h, j] * np.log(intensity_matrix[h, j])
        
        return np.exp(log_likelihood)
```

```python
"""
core/representation.py - Représentations versatiles de processus ponctuels
Référence: versatile_pp.tex Section 2, uniform_temp_coding.tex Proposition 1
"""

from abc import ABC, abstractmethod
from typing import List, Tuple, Optional
import numpy as np
from enum import Enum

class RepresentationType(Enum):
    TIMESTAMP = 0
    COUNT = 1
    INTERVAL = 2
    BOOLEAN = 3
    COMBINATORIAL = 4

class Representation(ABC):
    """Classe de base pour les représentations"""
    
    def __init__(self, r_bins: int, rep_type: RepresentationType):
        self.r_bins = r_bins
        self.rep_type = rep_type
        self.data = None
    
    @abstractmethod
    def from_jump_times(self, jump_times: List[float], time_interval: Tuple[float, float]):
        """Transforme les temps de sauts en représentation"""
        pass
    
    @abstractmethod
    def to_jump_times(self, time_interval: Tuple[float, float]) -> List[float]:
        """Reconstruit les temps de sauts depuis la représentation"""
        pass
    
    @abstractmethod
    def code_length(self, n_jumps: int, m_colors: int) -> float:
        """Calcule la longueur de code pour cette représentation"""
        pass
    
    def get_jump_times(self) -> Optional[List[float]]:
        """Primitive d'accès aux temps de sauts"""
        return self.to_jump_times((0, self.r_bins)) if hasattr(self, '_time_interval') else None
    
    def has_jump_at(self, t: int) -> bool:
        """Primitive: vérifie présence de saut au temps t"""
        if self.rep_type == RepresentationType.BOOLEAN:
            return self.data[t] == 1
        return False
    
    def get_count_in_bin(self, b: int) -> int:
        """Primitive: retourne le nombre de sauts dans le bin b"""
        if self.rep_type == RepresentationType.COUNT:
            return self.data[b]
        return 0

class TimestampRepresentation(Representation):
    """Représentation par temps de sauts exacts"""
    
    def __init__(self, r_bins: int):
        super().__init__(r_bins, RepresentationType.TIMESTAMP)
    
    def from_jump_times(self, jump_times: List[float], time_interval: Tuple[float, float]):
        self.data = sorted(jump_times)
        self._time_interval = time_interval
    
    def to_jump_times(self, time_interval: Tuple[float, float]) -> List[float]:
        return self.data if self.data else []
    
    def code_length(self, n_jumps: int, m_colors: int) -> float:
        """L = log(N) + N*log(r) + N*log(m)"""
        if n_jumps == 0:
            return 0
        return np.log2(n_jumps) + n_jumps * np.log2(self.r_bins) + n_jumps * np.log2(m_colors)

class CountRepresentation(Representation):
    """Représentation par histogramme de comptes"""
    
    def __init__(self, r_bins: int):
        super().__init__(r_bins, RepresentationType.COUNT)
    
    def from_jump_times(self, jump_times: List[float], time_interval: Tuple[float, float]):
        self.data = np.zeros(self.r_bins, dtype=int)
        dt = (time_interval[1] - time_interval[0]) / self.r_bins
        for t in jump_times:
            bin_idx = min(int((t - time_interval[0]) / dt), self.r_bins - 1)
            self.data[bin_idx] += 1
    
    def to_jump_times(self, time_interval: Tuple[float, float]) -> List[float]:
        # Reconstruction approximative (milieu du bin)
        dt = (time_interval[1] - time_interval[0]) / self.r_bins
        jump_times = []
        for i, count in enumerate(self.data):
            for _ in range(count):
                jump_times.append(time_interval[0] + (i + 0.5) * dt)
        return jump_times
    
    def code_length(self, n_jumps: int, m_colors: int) -> float:
        """L = r + N*log(m)"""
        return self.r_bins + n_jumps * np.log2(m_colors)

class IntervalRepresentation(Representation):
    """Représentation par intervalles entre sauts"""
    
    def __init__(self, r_bins: int):
        super().__init__(r_bins, RepresentationType.INTERVAL)
    
    def from_jump_times(self, jump_times: List[float], time_interval: Tuple[float, float]):
        sorted_times = sorted(jump_times)
        if len(sorted_times) > 0:
            self.data = [sorted_times[0]] + [
                sorted_times[i] - sorted_times[i-1] 
                for i in range(1, len(sorted_times))
            ]
        else:
            self.data = []
    
    def to_jump_times(self, time_interval: Tuple[float, float]) -> List[float]:
        if not self.
            return []
        jump_times = [self.data[0]]
        for interval in self.data[1:]:
            jump_times.append(jump_times[-1] + interval)
        return jump_times
    
    def code_length(self, n_jumps: int, m_colors: int) -> float:
        """L = N*log(r) + N*log(m)"""
        if n_jumps == 0:
            return 0
        return n_jumps * np.log2(self.r_bins) + n_jumps * np.log2(m_colors)

class BooleanRepresentation(Representation):
    """Représentation par vecteur booléen"""
    
    def __init__(self, r_bins: int):
        super().__init__(r_bins, RepresentationType.BOOLEAN)
    
    def from_jump_times(self, jump_times: List[float], time_interval: Tuple[float, float]):
        self.data = np.zeros(self.r_bins, dtype=int)
        dt = (time_interval[1] - time_interval[0]) / self.r_bins
        for t in jump_times:
            bin_idx = min(int((t - time_interval[0]) / dt), self.r_bins - 1)
            self.data[bin_idx] = 1
    
    def to_jump_times(self, time_interval: Tuple[float, float]) -> List[float]:
        dt = (time_interval[1] - time_interval[0]) / self.r_bins
        return [time_interval[0] + (i + 0.5) * dt 
                for i in range(self.r_bins) if self.data[i] == 1]
    
    def code_length(self, n_jumps: int, m_colors: int) -> float:
        """L = r + N*log(m)"""
        return self.r_bins + n_jumps * np.log2(m_colors)

class CombinatorialRepresentation(Representation):
    """Représentation par adresse combinatoire (N + Index)"""
    
    def __init__(self, r_bins: int):
        super().__init__(r_bins, RepresentationType.COMBINATORIAL)
        self.n_jumps = 0
        self.config_index = 0
    
    def from_jump_times(self, jump_times: List[float], time_interval: Tuple[float, float]):
        # Créer d'abord le vecteur booléen
        bool_rep = BooleanRepresentation(self.r_bins)
        bool_rep.from_jump_times(jump_times, time_interval)
        
        self.data = bool_rep.data
        self.n_jumps = sum(self.data)
        self.config_index = self._compute_combinatorial_index(self.data, self.n_jumps)
    
    def _compute_combinatorial_index(self, bool_vector: np.ndarray, n_ones: int) -> int:
        """Calcule l'index combinatoire parmi C_r^N configurations"""
        from math import comb
        index = 0
        ones_seen = 0
        for i, bit in enumerate(bool_vector):
            if bit == 1:
                ones_seen += 1
            else:
                if ones_seen < n_ones:
                    index += comb(self.r_bins - i - 1, n_ones - ones_seen)
        return index
    
    def to_jump_times(self, time_interval: Tuple[float, float]) -> List[float]:
        bool_rep = BooleanRepresentation(self.r_bins)
        bool_rep.data = self.data
        return bool_rep.to_jump_times(time_interval)
    
    def code_length(self, n_jumps: int, m_colors: int) -> float:
        """L = log(N) + log(C_r^N) + N*log(m)"""
        from math import comb, log2
        if n_jumps == 0:
            return 0
        return log2(n_jumps) + log2(comb(self.r_bins, n_jumps)) + n_jumps * log2(m_colors)
```

```python
"""
compression/uniform.py - Codage uniforme
Référence: uniform_temp_coding.tex Proposition 1
"""

import numpy as np
from math import comb, log2
from typing import Tuple

class UniformCoder:
    """Codage uniforme avec les 4 méthodes L1-L4"""
    
    def __init__(self, r: int, m: int):
        self.r = r  # Nombre de bins temporels
        self.m = m  # Nombre de couleurs/états
    
    def L1_full_state(self) -> float:
        """L1 = r * log(m) - État complet"""
        return self.r * log2(self.m)
    
    def L2_list_jumps(self, N: int) -> float:
        """L2 = log(N) + N * log(r*m) - Liste des sauts"""
        if N == 0:
            return 0
        return log2(N) + N * log2(self.r * self.m)
    
    def L3_boolean(self, N: int) -> float:
        """L3 = r + N * log(m) - Vecteur booléen"""
        return self.r + N * log2(self.m)
    
    def L4_combinatorial(self, N: int) -> float:
        """L4 = log(N) + log(C_r^N) + N * log(m) - Adresse combinatoire"""
        if N == 0:
            return 0
        return log2(N) + log2(comb(self.r, N)) + N * log2(self.m)
    
    def optimal_length(self, N: int) -> Tuple[float, int]:
        """Retourne la longueur minimale et la méthode optimale (1-4)"""
        lengths = [
            self.L1_full_state(),
            self.L2_list_jumps(N),
            self.L3_boolean(N),
            self.L4_combinatorial(N)
        ]
        min_idx = np.argmin(lengths)
        return lengths[min_idx], min_idx + 1
    
    def get_thresholds(self) -> dict:
        """Calcule les seuils de décision δ_r,m, γ_34"""
        delta_r_m = np.ceil(self.r * (1 - 1 / log2(self.m)))
        return {
            'delta_r_m': delta_r_m,
            'r': self.r,
            'm': self.m
        }
```

```python
"""
compression/universal.py - Codage universel sur les entiers
Référence: uniform_temp_coding.tex Section 2.2
"""

class UniversalCoder:
    """Codes universels pour entiers (Elias, Rissanen)"""
    
    @staticmethod
    def elias_gamma(n: int) -> str:
        """Code Elias gamma"""
        if n == 0:
            return '0'
        binary = bin(n)[2:]
        return '0' * (len(binary) - 1) + binary
    
    @staticmethod
    def elias_delta(n: int) -> str:
        """Code Elias delta"""
        if n == 0:
            return '0'
        binary = bin(n)[2:]
        length_code = UniversalCoder.elias_gamma(len(binary) - 1)
        return length_code + binary[1:]
    
    @staticmethod
    def rissanen_code(n: int) -> str:
        """Code de Rissanen (asymptotiquement optimal)"""
        if n <= 0:
            return '0'
        # L(n) ≈ log(n) + 2*log(log(n))
        code = UniversalCoder.elias_delta(n)
        return code
    
    @staticmethod
    def code_length(n: int) -> float:
        """Longueur approximative du code universel"""
        if n <= 0:
            return 1
        return np.log2(n) + 2 * np.log2(np.log2(n + 1)) + 1
```

```python
"""
compression/mdl.py - Codage statistique MDL
Référence: versatile_main.tex Théorème 1
"""

import numpy as np
from typing import List, Tuple
from core.intensity import IntensityFamily

class MDLCoder:
    """Codage statistique avec principe MDL"""
    
    def __init__(self, candidate_families: List[IntensityFamily]):
        self.candidate_families = candidate_families
        self.selected_family = None
        self.estimated_intensity = None
    
    def estimate_intensity(self, jump_times: List[float], time_interval: Tuple[float, float]):
        """
        Estime la fonction d'intensité par MDL
        Référence: versatile_main.tex Eq. (1)
        """
        best_mdl = float('inf')
        best_family = None
        best_intensity = None
        
        for family in self.candidate_families:
            # Estimer les paramètres
            intensity_func = family.fit(jump_times, time_interval)
            
            # Calculer la vraisemblance
            log_likelihood = sum(np.log(intensity_func(t)) for t in jump_times)
            integral = family.numerical_integral(intensity_func, time_interval[0], time_interval[1])
            log_likelihood -= integral
            
            # Calculer la complexité
            complexity = family.complexity_penalty()
            
            # Critère MDL
            mdl_score = -log_likelihood + complexity
            
            if mdl_score < best_mdl:
                best_mdl = mdl_score
                best_family = family
                best_intensity = intensity_func
        
        self.selected_family = best_family
        self.estimated_intensity = best_intensity
        return best_intensity
    
    def code_length(self, jump_times: List[float], time_interval: Tuple[float, float]) -> float:
        """
        Calcule la longueur de code MDL
        Référence: versatile_main.tex Thm 1
        """
        if self.estimated_intensity is None:
            self.estimate_intensity(jump_times, time_interval)
        
        # Terme de vraisemblance
        log_likelihood = sum(np.log(self.estimated_intensity(t)) for t in jump_times)
        integral = self.selected_family.numerical_integral(
            self.estimated_intensity, 
            time_interval[0], 
            time_interval[1]
        )
        log_likelihood -= integral
        
        # Terme de complexité
        complexity = self.selected_family.complexity_penalty()
        
        return -log_likelihood + complexity
```

```python
color/mode.py - Modes de gestion des couleurs
Référence: uniform_temp_coding.tex Proposition 1

from enum import Enum
from typing import List, Dict
import numpy as np

class ColorModeType(Enum):
    MARKED = 0
    STATE_BY_STATE = 1

class ColorMode:
    """Gestion du codage des couleurs"""
    
    def __init__(self, mode_type: ColorModeType, n_colors: int):
        self.mode_type = mode_type
        self.n_colors = n_colors
        self.color_map: Dict[int, int] = {}  # Couleur globale -> Couleur locale
    
    def color_cost(self, n_jumps: int) -> float:
        """
        Calcule le terme de coût pour les couleurs
        Référence: uniform_temp_coding.tex L2, L3, L4
        """
        if self.mode_type == ColorModeType.MARKED:
            # Coût par saut: N * log(m)
            return n_jumps * np.log2(self.n_colors)
        else:
            # Coût par bloc: C_map (codé une fois)
            return self._color_map_cost()
    
    def _color_map_cost(self) -> float:
        """Coût de la table de correspondance des couleurs"""
        # Codage de m couleurs parmi l'espace global
        # Approximation: m * log(|E_global|)
        global_color_space = 256  # Ex: 256 couleurs possibles
        return self.n_colors * np.log2(global_color_space)
    
    def set_color_map(self, color_mapping: Dict[int, int]):
        """Définit la table de correspondance"""
        self.color_map = color_mapping
    
    def get_local_color(self, global_color: int) -> int:
        """Retourne la couleur locale pour une couleur globale"""
        return self.color_map.get(global_color, global_color)
```

```python
import numpy as np

class DecisionHeuristic:

        self.model = RandomForestClassifier(n_estimators=100, random_state=42)
        self.is_trained = False
    
    def extract_features(self, block_data: Dict) -> np.ndarray:
        """
        Extrait les caractéristiques d'un bloc vidéo
        """
        features = [
            block_data.get('n_jumps', 0),
            block_data.get('r_bins', 256),
            block_data.get('n_colors', 16),
            block_data.get('lambda_avg', 0),  # N/r

            block_data.get('cost_marked', 0),    # N*log(m)
            block_data.get('cost_state', 0),     # C_map
        ]
        return np.array(features).reshape(1, -1)
    
    def train(self, training_data: List[Dict], labels: List[int]):
        """
        Entraîne le modèle sur les données de codage exhaustif
        """
        X = np.vstack([self.extract_features(d) for d in training_data])
        y = np.array(labels)
        self.model.fit(X, y)
        self.is_trained = True
    
    def predict_method(self, block_ Dict) -> int:
        """
        Prédit la méthode optimale pour un bloc
        Retourne le code méthode (7 bits)
        """
        if not self.is_trained:
            raise ValueError("Modèle non entraîné")
        
        features = self.extract_features(block_data)
        prediction = self.model.predict(features)[0]
        return prediction
    
    def save(self, filepath: str):
```

#### 2.2.6. Gestion du Bitstream (`utils/bitstream.py`)

```python
"""
utils/bitstream.py - Gestion du flux binaire
"""

class BitStream:
    """Gestion du flux binaire pour le codage/décodage"""
    
    def __init__(self):
        self.bits = []
        self.position = 0
    
    def write_bits(self, value: int, n_bits: int):
        """Écrit n bits"""
        for i in range(n_bits - 1, -1, -1):
            self.bits.append((value >> i) & 1)
    
    def write_method_code(self, proc_type: int, rep_type: int, comp_type: int, color_mode: int):
        """
        Écrit le code méthode (7 bits)
        Structure: [PROC:2][REP:2][COMP:2][COL:1]
        """
        method_code = (proc_type << 5) | (rep_type << 3) | (comp_type << 1) | color_mode
        self.write_bits(method_code, 7)
    
    def read_method_code(self) -> Dict:
        """Lit le code méthode"""
        code = 0
        for i in range(7):
            if self.position + i < len(self.bits):
                code = (code << 1) | self.bits[self.position + i]
        self.position += 7
        
        return {
            'proc_type': (code >> 5) & 0b11,
            'rep_type': (code >> 3) & 0b11,
            'comp_type': (code >> 1) & 0b11,
            'color_mode': code & 0b1
        }
    
    def to_bytes(self) -> bytes:
        """Convertit en bytes"""
        byte_array = bytearray()
        for i in range(0, len(self.bits), 8):
            byte = 0
            for j in range(8):
                if i + j < len(self.bits):
                    byte = (byte << 1) | self.bits[i + j]
                else:
                    byte = byte << 1
            byte_array.append(byte)
        return bytes(byte_array)
```

```python
"""
examples/demo.py - Démonstration du système de codage

from core.process import RealMarkedProcess, MonochromaticProcess

from compression.uniform import UniformCoder
from compression.mdl import MDLCoder

from color.mode import ColorMode, ColorModeType
from utils.bitstream import BitStream

def demo_video_coding():
    """Démonstration complète du codage vidéo"""
    
    # 1. Création d'un processus ponctuel (historique d'un pixel)
    time_interval = (0, 256)
    jump_times = [10, 45, 89, 134, 178, 223]
    colors = [1, 3, 2, 4, 1, 3]
    
    # 2. Choix du mode couleur
    n_colors = 4
    color_mode = ColorMode(ColorModeType.STATE_BY_STATE, n_colors)

    mono_processes = {}
    for c in range(n_colors):

        mono_processes[m].add_jump(t)

    r_bins = 256
    rep = CombinatorialRepresentation(r_bins)
    rep.from_jump_times(jump_times, time_interval)

    color_cost = color_mode.color_cost(N)
    L_total_uniform = L_uniform + color_cost
    L_total_mdl = L_mdl + color_cost
    
    print(f"Longueur totale uniforme: {L_total_uniform:.2f} bits")
    print(f"Longueur totale MDL: {L_total_mdl:.2f} bits")
    
    # 8. Sélection de la méthode
    if L_total_mdl < L_total_uniform:
        selected_method = 'MDL'
        selected_length = L_total_mdl
    else:
        selected_method = 'UNIFORM'
        selected_length = L_total_uniform
    
    print(f"Méthode sélectionnée: {selected_method}")
    
    # 9. Création du bitstream
    bitstream = BitStream()

        'uniform_length': L_total_uniform,
        'mdl_length': L_total_mdl,
        'selected_method': selected_method,
        'compression_ratio': (N * np.log2(r_bins * n_colors)) / selected_length
```

# Fonctionnalités Avancées basées sur la Structure des Processus Ponctuels

## 1. Principe Fondamental : Préservation de l'Information Stochastique
Conformément à `versatile_main.tex` et `uniform_temp_coding.tex`, le codage repose sur la description des sauts ($\tau_k$) et des états ($m_k$).
*   **Sans perte :** La reconstruction est exacte ($X_{original} = X_{reconstruit}$).
*   **Accès direct :** Les paramètres du modèle (intensité $\alpha(t)$, nombre de sauts $N$, carte de couleurs) sont des métadonnées du flux compressé.
*   **Manipulation :** On peut modifier ces métadonnées pour altérer la vidéo reconstruite (ex: changer une couleur, supprimer une zone de faible intensité) avant même le décodage spatial.

## 2. Fonctionnalités basées sur le Pré-Codage (Structure Brute)
Ces fonctionnalités exploitent les données discrétisées avant l'estimation statistique fine. Elles correspondent aux représentations versatiles de `versatile_pp.tex`.

### 2.1. Détection d'Événements Temporels (Event Detection)
*   **Principe :** Utiliser la représentation **Boolean** ou **Timestamp**.
*   **Fonctionnalité :** Identifier les instants précis où un changement sémantique occur (ex: flash, coupure, mouvement brusque).
*   **Implémentation :**
    *   Si $N_i > Seuil$ pour un bloc $i$, alors "Événement détecté".
    *   Référence : `Uniform_temporal.tex` (Modélisation du nombre de sauts).
*   **Usage :** Indexation vidéo, résumé automatique (keyframes aux sauts d'intensité).

### 2.2. Quantification et Filtrage des Couleurs (Color Filtering)
*   **Principe :** Exploiter la décomposition monochromatique (Processus par couleur).
*   **Fonctionnalité :** Supprimer ou isoler certaines couleurs sans décoder l'image.
*   **Implémentation :**
    *   **Mode État par État :** Chaque couleur $c$ est un processus $N^{(c)}$. Pour supprimer le rouge, on met à zéro le processus $N^{(rouge)}$.
    *   **Mode Marqué :** Filtrer la liste des marques $\{m_k\}$. Si $m_k = rouge$, on ignore le saut $\tau_k$.
    *   Référence : `uniform_temp_coding.tex` (Termes de coût $N \log m$).
*   **Usage :** Accessibilité (daltonisme), confidentialité (flouter des couleurs spécifiques), compression sélective.

### 2.3. Rééchantillonnage Temporel (Temporal Resampling)
*   **Principe :** Utiliser la représentation **Interval** ou **Timestamp**.
*   **Fonctionnalité :** Accélérer ou ralentir la vidéo localement.
*   **Implémentation :**
    *   Multiplier les intervalles $\Delta \tau_k$ par un facteur $\beta$.
    *   Si $\beta > 1$ (Ralentis sur les zones de mouvement).
    *   Référence : `versatile_pp.tex` (Transformation entre représentations).

## 3. Fonctionnalités basées sur la Compression (Modèles Statistiques)
Ces fonctionnalités exploitent les paramètres estimés lors de la phase MDL (`versatile_main.tex`).

### 3.1. Carte de Saillance par Intensité (Intensity Saliency Map)
*   **Principe :** La fonction d'intensité estimée $\hat{\alpha}(t)$ représente la probabilité de changement par unité de temps.
*   **Fonctionnalité :** Identifier les zones d'intérêt (ROI) dynamiques.
*   **Implémentation :**
    *   Calculer l'intensité moyenne par bloc spatial : $\bar{\alpha}_i = \frac{1}{T} \int_0^T \hat{\alpha}_i(t) dt$.
    *   Les blocs avec $\bar{\alpha}_i > \lambda_{seuil}$ sont des zones d'activité (mouvement, changement de texture).
    *   Référence : `versatile_main.tex` (Estimation de l'intensité d'un processus de Poisson).
*   **Usage :** Compression adaptative (allouer plus de bits aux zones actives), surveillance (détection d'intrusion).

### 3.2. Segmentation Spatio-Temporelle par Modèle
*   **Principe :** Le choix du modèle MDL (Histogramme vs Splines vs Ondelettes) indique la nature du signal.
*   **Fonctionnalité :** Classifier les types de mouvement.
*   **Implémentation :**
    *   **Modèle Constant (Histogramme 1 bin) :** Zone statique ou bruit de fond.
    *   **Modèle Lisse (Splines) :** Mouvement régulier (objet en translation).
    *   **Modèle Complexe (Ondelettes) :** Mouvement chaotique ou texture changeante.
    *   Référence : `versatile_main.tex` (Sélection de modèle par pénalité de complexité).
*   **Usage :** Analyse de comportement, classification de scènes.

### 3.3. Modification de la Dynamique des Changements
*   **Principe :** Modifier la fonction d'intensité cumulée utilisée pour le codage arithmétique.
*   **Fonctionnalité :** Lisser ou accentuer les transitions temporelles.
*   **Implémentation :**
    *   Appliquer un filtre passe-bas sur $\hat{\alpha}(t)$ avant le décodage.
    *   Résultat : Les sauts trop rapprochés sont fusionnés ou supprimés (réduction du bruit temporel).
    *   Référence : `Uniform_temporal.tex` (Modèle multiplicatif d'intensité).

```python
        return [b for b in self.stream.blocks if b.intensity_avg > threshold]

        for block in self.stream.blocks:
```

# Optimisation par Représentation Binaire Spatiale et Opérations Bitwise

## 1. Formalisation Mathématique

### 1.1. Masques Binaires Monochromes
Soit une vidéo discrétisée spatialement en pixels $p \in \mathcal{S}$ et temporellement en frames $t \in \{1, \dots, T\}$.
Pour chaque couleur $c \in \{1, \dots, m\}$, on définit un **champ binaire spatial** (ou masque) $M_c(t)$ à l'instant $t$ :

$$ M_c(t, p) = \begin{cases} 
1 & \text{si le pixel } p \text{ a la couleur } c \text{ à l'instant } t \\
0 & \text{sinon}
\end{cases} $$

Cela correspond à la **représentation binaire** d'un processus ponctuel spatial vue dans `versatile_pp.tex`, où l'événement est la "présence de la couleur $c$".

### 1.2. Détection de Sauts par XOR
Un saut (changement d'état) pour la couleur $c$ au pixel $p$ entre $t$ et $t+1$ est défini par :
$$ \text{Saut}_c(t, p) = 1 \iff M_c(t, p) \neq M_c(t+1, p) $$

Cette opération logique est exactement un **OU Exclusif (XOR)** bit à bit :
$$ \Delta_c(t) = M_c(t) \oplus M_c(t+1) $$

où $\Delta_c(t)$ est la carte des changements pour la couleur $c$.

### 1.3. Comptage des Sauts ($N$)
Le nombre total de sauts $N_c$ pour la couleur $c$ sur l'intervalle $[0, T]$ est la somme des bits activés dans les cartes de changement (norme $L_0$ ou *Population Count*) :
$$ N_c = \sum_{t=0}^{T-1} \text{popcount}(\Delta_c(t)) $$

Ce comptage est essentiel pour appliquer les critères de sélection de codage ($L_1$ vs $L_3$ vs $L_4$) de la **Proposition 1 de `uniform_temp_coding.tex`**.

---

## 2. Impact sur l'Architecture de Codage

Cette approche transforme des opérations de comparaison d'entiers (coûteuses) en opérations vectorielles sur les registres CPU/GPU (extrêmement rapides).

### 2.1. Module d'Extraction Accéléré
| Fonctionnalité | Méthode Classique | Méthode Optimisée (Bitwise) | Gain |
| :--- | :--- | :--- | :--- |
| **Comparaison** | `if (pixel[t] != pixel[t+1])` | `XOR(Register[t], Register[t+1])` | x64 (SIMD) |
| **Comptage Sauts** | Boucle incrémentale | Instruction `POPCOUNT` | x10 |
| **Localisation** | Parcours liste | Instruction `TZCNT` (Trail Zero Count) | Variable |
| **Mémoire** | Stockage valeurs couleurs | Stockage bits (1/8ème de la taille) | x8 |

### 2.2. Intégration dans le Cahier des Charges
Il faut ajouter une contrainte d'alignement mémoire pour profiter des instructions SIMD (SSE, AVX, NEON).

*   **Structure de Données :** Les masques $M_c(t)$ doivent être alignés sur des mots de 64 bits (ou 128/256 bits).
*   **Primitives CPU :** Utiliser les intrinsèques compilateur (`_mm_xor_si128`, `_mm_popcnt_u64`).

## 3. Implications sur les Longueurs de Code (`uniform_temp_coding.tex`)

Cette représentation favorise naturellement certains modes de codage décrits dans la **Proposition 1** :

### 3.1. Favorisation du Codage Booléen ($L_3$)
La formule $L_3 = r + N \log m$ devient très compétitive car :
1.  Le terme $r$ (nombre de bins temporels) correspond directement à la structure de frames.
2.  Le vecteur booléen est déjà construit nativement par l'opération XOR.
3.  Si l'activité est dense (beaucoup de changements), $L_3$ est souvent optimal selon la **Proposition 1 (point 2)**.

### 3.2. Favorisation du Codage Combinatoire ($L_4$)
La formule $L_4 = \log N + \log C_r^N + N \log m$ est pertinente si $N$ est faible.
*   Grâce au `POPCOUNT`, on obtient $N$ instantanément.
*   Si $N \ll r$, on passe en mode codage d'adresse (liste des positions des bits à 1 dans le XOR).

### 3.3. Élimination du Terme $N \log m$ (Mode État par État)
Comme discuté précédemment, en traitant chaque couleur $c$ comme un processus indépendant :
*   On code la position des bits à 1 dans $\Delta_c$.
*   La couleur est implicite (c'est l'indice du processus $c$).
*   Le terme $N \log m$ disparaît du codage temporel, remplacé par un coût fixe d'en-tête de bloc pour la palette.

## 5. Exemple d'Implémentation (C++ / Intrinsèques)

Voici comment cela se traduit dans le module d'extraction :

```cpp
#include <immintrin.h> // Pour AVX2

// Structure alignée pour les masques binaires
struct alignas(32) BinaryMask {
    uint64_t data[NUM_PIXELS / 64];
};

// Détection de sauts et comptage N pour une couleur c
uint64_t extract_jumps_and_count(const BinaryMask& frame_t, 
                                 const BinaryMask& frame_t1, 
                                 std::vector<int>& jump_indices) {
    uint64_t total_jumps = 0;
    
    // Parcours vectoriel (SIMD)
    for (int i = 0; i < NUM_PIXELS / 64; ++i) {
        // 1. XOR pour trouver les différences
        uint64_t diff = frame_t.data[i] ^ frame_t1.data[i];
        
        // 2. POPCOUNT pour compter les sauts (N)
        int jumps = _mm_popcnt_u64(diff);
        total_jumps += jumps;
        
        // 3. Si besoin de localisation précise (pour TimeStamps)
        if (jumps > 0 && !jump_indices.empty()) {
            while (diff) {
                int bit = _tzcnt_u64(diff); // Count Trailing Zeros
                jump_indices.push_back(i * 64 + bit);
                diff &= diff - 1; // Clear lowest set bit
            }
        }
    }
    return total_jumps;
}
```

## 1. Fonctionnalité 1 : Organisation Hiérarchique des Couleurs (Principe des États Multi-Niveaux)

### 1.1. Modèle Mathématique

Conformément au modèle de processus ponctuel marqué de `uniform_temp_coding.tex` (Section 2.1), l'espace des marques $\mathcal{E}$ est étendu pour supporter une structure hiérarchique :

$$\mathcal{E} = \mathcal{E}_1 \times \mathcal{E}_2 \times \mathcal{E}_3$$

où :
- $\mathcal{E}_1$ : Niveau 1 - Couleur de base (256 valeurs, 8 bits)
- $\mathcal{E}_2$ : Niveau 2 - Nuance (256 valeurs, 8 bits)
- $\mathcal{E}_3$ : Niveau 3 - Finesse/Texture (256 valeurs, 8 bits)

**Marque Hiérarchique :** $m = (m_1, m_2, m_3) \in \mathcal{E}$

### 1.2. Avantages du Modèle

| Niveau | Précision | Coût de Codage | Usage |
|--------|-----------|----------------|-------|
| **Niveau 1** | 256 couleurs | $\log(256) = 8$ bits | Aperçu, faible bande passante |
| **Niveau 1-2** | 65 536 nuances | $16$ bits | Qualité standard HD |
| **Niveau 1-3** | 16 777 216 couleurs | $24$ bits | Haute définition, HDR |

### 1.3. Implémentation dans le Codage

```python
class HierarchicalColor:
    def __init__(self, level1, level2=None, level3=None):
        self.level1 = level1  # 0-255
        self.level2 = level2  # 0-255 (optionnel)
        self.level3 = level3  # 0-255 (optionnel)
        self.precision_level = sum([1 if x else 0 for x in [level1, level2, level3]])
    
    def code_length(self, target_level=3):
        """
        Calcule la longueur de code selon le niveau de précision cible
        Référence: uniform_temp_coding.tex Proposition 1 (terme N*log(m))
        """
        bits_per_level = 8
        return self.precision_level * bits_per_level
    
    def downgrade(self, target_level):
        """Réduit la précision chromatique selon les besoins"""
        if target_level == 1:
            return HierarchicalColor(self.level1)
        elif target_level == 2:
            return HierarchicalColor(self.level1, self.level2)
        return self
```

### 1.4. Impact sur les Longueurs de Code

Les formules de `uniform_temp_coding.tex` (Proposition 1) sont adaptées :

- **L1 (État complet) :** $L_1 = r \times \sum_{i=1}^{L} \log(m_i)$
- **L2 (Liste sauts) :** $L_2 = \log(N) + N \times (\log(r) + \sum_{i=1}^{L} \log(m_i))$
- **L3 (Booléen) :** $L_3 = r + N \times \sum_{i=1}^{L} \log(m_i)$
- **L4 (Combinatoire) :** $L_4 = \log(N) + \log(C_r^N) + N \times \sum_{i=1}^{L} \log(m_i)$

où $L$ est le nombre de niveaux de précision actifs.

## 2. Fonctionnalité 2 : Résolution Spatiale Dynamique (Arbre de Processus)

### 2.1. Modèle Mathématique

Conformément à `versatile_main.tex` (Section 4.2 - Estimation of a Poisson process measure), chaque processus spatial $N_s$ peut avoir des processus enfants $N_{s,c}$ représentant des détails :

$$N_s = \sum_{c \in Children(s)} N_{s,c}$$

**Structure Arborescente :**
```
Racine (Basse résolution)
├── Enfant 1 (Détail zone 1)
│   ├── Petit-enfant 1.1 (Détail fin)
│   └── Petit-enfant 1.2 (Détail fin)
├── Enfant 2 (Détail zone 2)
└── Enfant 3 (Détail zone 3)
```

### 2.2. Spécifications Techniques

| Niveau | Résolution | Nombre de Processus | Usage |
|--------|------------|---------------------|-------|
| **Niveau 0** | 16x16 pixels | 1 processus/bloc | Aperçu rapide |
| **Niveau 1** | 8x8 pixels | 4 processus/bloc | Qualité standard |
| **Niveau 2** | 4x4 pixels | 16 processus/bloc | Haute définition |
| **Niveau 3** | 2x2 pixels | 64 processus/bloc | Ultra HD |

### 2.3. Implémentation de l'Arbre

```python
class SpatialProcessTree:
    def __init__(self, level=0, coordinates=(0,0), size=16):
        self.level = level
        self.coordinates = coordinates
        self.size = size
        self.point_process = None
        self.children = []  # Liste de SpatialProcessTree
        self.parent = None
    
    def subdivide(self):
        """Divise le processus en 4 enfants (quadtree)"""
        if self.level >= 3:
            return  # Niveau maximum atteint
        
        half_size = self.size // 2
        offsets = [(0,0), (0,1), (1,0), (1,1)]
        
        for i, (dx, dy) in enumerate(offsets):
            child = SpatialProcessTree(
                level=self.level + 1,
                coordinates=(self.coordinates[0] + dx * half_size,
                           self.coordinates[1] + dy * half_size),
                size=half_size
            )
            child.parent = self
            self.children.append(child)
    
    def get_processes_at_level(self, target_level):
        """Récupère tous les processus à un niveau de résolution donné"""
        if self.level == target_level:
            return [self]
        elif self.level < target_level and self.children:
            processes = []
            for child in self.children:
                processes.extend(child.get_processes_at_level(target_level))
            return processes
        return []
    
    def code_length(self, active_levels):
        """
        Calcule la longueur de code pour les niveaux actifs
        Référence: versatile_main.tex (complexité C_n(Q))
        """
        total_length = 0
        if self.level in active_levels and self.point_process:
            total_length += self.point_process.code_length()
        for child in self.children:
            total_length += child.code_length(active_levels)
        return total_length
```

### 5.2. Moteur de Requêtes

```python
class PPVQueryEngine:
    def __init__(self, video_data):
        self.video_data = video_data
        self.spatial_index = None
        self.temporal_index = None
    
    def execute(self, query_string):
        """Exécute une requête PPV-QL"""
        parsed = self.parse_query(query_string)
        
        # Filtrage temporel
        if parsed.temporal_condition:
            results = self.filter_temporal(
                parsed.temporal_condition
            )
        
        # Filtrage spatial
        if parsed.spatial_condition:
            results = self.filter_spatial(
                results, parsed.spatial_condition
            )
        
        # Filtrage par intensité/couleur
        if parsed.value_condition:
            results = self.filter_values(
                results, parsed.value_condition
            )
        
        return results
    
    def filter_temporal(self, condition):
        """
        Filtre basé sur les temps de sauts
        Référence: Uniform_temporal.tex (modèle d'intensité)
        """
        results = []
        for block in self.video_data.blocks:
            jump_times = block.process.get_jump_times()
            if any(condition.check(t) for t in jump_times):
                results.append(block)
        return results
    
    def filter_spatial(self, blocks, region):
        """Filtre basé sur la région spatiale"""
        if not self.spatial_index:
            self.build_spatial_index()
        return self.spatial_index.query(region, blocks)
    
    def filter_values(self, blocks, condition):
        """Filtre basé sur les valeurs (couleur, intensité)"""
        results = []
        for block in blocks:
            if condition.evaluate(block.process):
                results.append(block)
        return results
```

| **Spatial** | Quadtree/R-Tree | $O(\log S)$ | Requêtes par région |
| **Intensity** | Histogramme cumulé | $O(1)$ | Seuil d'activité |

```python
class CacheManager:
    def __init__(self, max_memory_mb=512):
        self.max_memory = max_memory_mb * 1024 * 1024
        self.current_usage = 0
        self.cache = {}  # level -> data
    
    def preload_levels(self, video_id, levels=[0, 1]):
        """Précharge les niveaux spécifiés"""
        for level in levels:
            data = self.fetch_level(video_id, level)
            self.cache[(video_id, level)] = data
            self.current_usage += len(data)
    
    def get_level(self, video_id, level):

        key = (video_id, level)

            # Chargement à la demande

            self.evict_if_needed()
            self.cache[key] = data
            return data
```

## 7. Intégration avec le Cadre Théorique Existant

### 7.1. Compatibilité MDL

Toutes les extensions respectent le principe MDL de `versatile_main.tex` :

$$\hat{P}_n = \arg \min_{Q \in \Gamma_n} \left( -\log Q(X^n) + C_n(Q) \right)$$

- **Couleurs hiérarchiques :** $C_n(Q)$ inclut le coût des niveaux de précision actifs
- **Arbre spatial :** $C_n(Q)$ inclut la complexité de la structure arborescente
- **Zoom :** Sélection dynamique du sous-ensemble optimal de processus

### 7.2. Bornes de Convergence

Conformément à `versatile_main.tex` (Théorème 1) :

$$H^2(P, \hat{P}_n) \leq \min_{Q \in \Gamma_n} \left[ H^2(P, Q) + \frac{1}{n} C_n(Q) \right]$$

Les extensions maintiennent cette borne en ajoutant des termes de complexité contrôlés :
- $C_{color} = L \times 8$ bits (L = niveaux de couleur)
- $C_{spatial} = \log(\text{nb\_enfants})$ par nœud
- $C_{zoom} = \text{niveau\_actif} \times \text{taille\_bloc}$
